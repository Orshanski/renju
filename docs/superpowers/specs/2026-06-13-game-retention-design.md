# Спека: организация и ретеншн партий пользователя

**Дата:** 2026-06-13 · **Тип:** feature (продуктовая логика; бэкенд + фронт).
**Источник истины дизайна:** `docs/superpowers/specs/2026-06-07-renju-design.md`.

## 1. Проблема и цель

Партии копятся быстро — и доигранные, и брошенные недоигранные. Неограниченный
рост плох; но тупое авто-удаление по правилу рискует стереть особенно красивые
игры. Цель: **каждый пользователь сам ограничивает рост по количеству, а избранные
партии защищены и не теряются.** Чистка — без фонового таймера.

## 2. Три раздела (персональные, у каждого пользователя)

Раздел партии выводится из её состояния (отдельную «категорию» сверх необходимого
не вводим). Партия всегда ровно в ОДНОМ разделе:

- **Текущие** — незавершённые (`status` не `is_finished`), не избранные. На главном экране.
- **Завершённые** — `status.is_finished` (победа/поражение/ничья), не избранные. Свой раздел.
- **Избранное** — помеченные пользователем (новый признак `favorite`). Свой раздел. **Вне лимитов.**

Приоритет при выводе раздела: `favorite` → Избранное; иначе `is_finished` → Завершённые;
иначе → Текущие.

## 3. Персональные настройки (лимиты по количеству)

Живут в персональных настройках пользователя — **таблицу `user_settings` заводит эта
фича** (см. §8; rj-dix позже добавит undo-поля аддитивно):

- `current_limit` — макс. число Текущих. По умолчанию **включён, 10**.
- `finished_limit` — макс. число Завершённых. По умолчанию **включён, 50**.
- Оба настраиваются пользователем; каждый можно **выключить** (тогда раздел без лимита).
- Значение лимита — **≥ 1** (0 не имеет смысла: партия создавалась бы и тут же
  вытеснялась; «не держать ничего» — это не лимит 0, а отдельная семантика, не вводим).
- **Избранное лимита не имеет.**

Дефолтные числа (10 / 50) — ориентир, крутятся в настройках, не критичны.

## 4. Вытеснение (по количеству, событийно, без таймера)

Чистка срабатывает в момент, когда раздел может превысить лимит — фонового
таймера/cron нет:

- **Текущие:** при **создании новой партии** — если число СУЩЕСТВУЮЩИХ Текущих у
  пользователя превысит `current_limit`, жёстко удаляется **самая давно не тронутая**
  Текущая (по `updated_at` — последняя активность = «брошенная»). Создаваемая партия —
  самая свежая, кандидатом на вытеснение не является (лимит считается по существующим).
- **Завершённые:** при **завершении партии** (переход в `is_finished`) — если число
  Завершённых превысит `finished_limit`, жёстко удаляется партия с **самым старым
  `finished_at`** (момент завершения — отдельное поле, см. §7).
- **Тай-брейк** при равном времени (массовое создание / грубость timestamp) — вторичный
  ключ `created_at`, затем `id`: вытеснение должно быть детерминированным (важно для тестов).
- **Изменение настройки — тоже событие чистки:** при включении лимита или его
  снижении, если раздел уже превышает новое значение, он **немедленно подрезается**
  до лимита (по тому же правилу старшинства). Повышение/выключение — ничего не удаляет.
- **Сколько вытесняем:** ровно столько, чтобы в разделе осталось `limit` (на создании
  Текущей / на финише — как правило одна; при снижении лимита — сколько нужно).
- Чистка одного раздела не трогает другие; Избранное не вытесняется никогда.
- Лимит выключен → вытеснения в этом разделе нет.

## 5. Избранное

- Пометить **«В избранное» можно только Завершённую** партию (недоигранную — нельзя).
- **«Из избранного»** → партия возвращается в Завершённые и снова подпадает под
  `finished_limit`; если он при этом окажется превышен — вытеснение по правилу §4.
- Избранное числом не ограничено.

## 6. Действия пользователя

По партии — контекстное действие (**long-tap** на тач / **правый клик** мышью):

- **«В избранное»** (на Завершённой) ⇄ **«Из избранного»** (на Избранной).
- **«Удалить»** — жёсткое удаление насовсем; доступно в любом разделе (в т.ч.
  «бросить» недоигранную Текущую).

## 7. Что затрагивается (ориентир для плана — реализацию здесь НЕ детализируем)

- **Данные:** новый признак `favorite` у партии (`Game` сейчас его не имеет); и
  **новое поле `finished_at`** для порядка вытеснения Завершённых. Семантика:
  ставится в момент **последнего** перехода партии в `is_finished`; при `undo` из
  `finished_*` (партия возвращается в Текущие, мастер-спека §4.8) — **сбрасывается в
  NULL** (доиграется второй раз — проставится заново). Снятие из избранного `finished_at`
  **НЕ трогает** — партия сохраняет исходный момент завершения (детерминизм вытеснения).
  `updated_at` для порядка завершения НЕ годится: `undo` из finished и снятие из
  избранного дёргают `onupdate` и сдвигают его.
- **Бэкенд:** эндпоинты `delete` / `favorite` / `unfavorite`; **отдельный лёгкий
  list-эндпоинт по разделам** — отдаёт summary-DTO (без полного состояния и тяжёлых
  вычислений); репозиторий сейчас — `list_by_owner` без категорий/summary и без
  `delete`, расширить; логика вытеснения на создании партии и на её завершении;
  персональные лимиты в `user_settings`.
- **Фронт:** три раздела (Текущие на главной, Завершённые, Избранное) + контекстные
  действия.

## 8. Тикет и сверка scope

Это **rj-as6** (фронт-срез 3: список / новая партия / восстановление) — НЕ новый тикет.
Сверка с его требованиями (из приёмки rj-p82):

- **Хранилище настроек (лимиты) + дефолты — в БД, в рамках этой работы.** Эта фича
  **заводит таблицу `user_settings`** (поля лимитов + флаги + миграция) — её ещё нет.
  rj-dix позже добавит свои поля (undo-политика) **аддитивно** в ту же таблицу (без
  коллизии миграций). Экран редактирования («морда») — отдельной задачей (rj-dix):
  дефолты в БД работают и без UI.
- **Разделы + действия по партиям** (Текущие/Завершённые/Избранное, long-tap / правый
  клик, удаление, избранное) — это и есть ядро «списка» rj-as6.
- **Лёгкий список — ОТДЕЛЬНЫЙ эндпоинт.** Список (три раздела) отдаёт **summary-DTO**
  на партию: id / статус / уровень / твой цвет / № хода / когда обновлено — БЕЗ полного
  состояния и тяжёлых вычислений (winning_line/cursor/forbidden). Это отдельный эндпоинт,
  не теперешний полный `GET /api/games/:id`. Нужен независимо от числа партий — не
  таскать полный стейт ради списка.
- **Пагинация** (в rj-as6 требовалась от роста до сотен) — сейчас **не строим**.
  Лимиты держат Текущие/Завершённые малыми; Избранное — **сознательно без лимита**
  (решение Alexey: «красивых» бережём, их по логике единицы-десятки), но раздел отдаёт
  тот же лёгкий summary-DTO, так что список дёшев даже подлиннее. Если Избранное
  когда-нибудь распухнет до сотен — пагинация **отдельным тикетом**. (Снятие
  пагинации, заявленной в rj-as6, — осознанное решение.)
- **Экран новой партии** (выбор уровня → создать; заменяет временную кнопку novice;
  `GET /api/levels` и `POST /api/games` уже есть) — соседняя простая задача того же
  тикета, отдельного дизайна не просит, войдёт в план rj-as6.
- **Навигация список↔партия / выход** — уже сделано ранее.
- **Срезы реализации:** план бьётся на **бэкенд-срез** (поля `favorite`/`finished_at`
  + миграция, логика вытеснения, эндпоинты delete/favorite/unfavorite + лёгкий list,
  лимиты в `user_settings`) → затем **фронт-срез** (три раздела + действия + экран
  новой партии).

## 9. Вне области задачи

- **Авто-удаление по возрасту** (партии старше T дней) — приносит фоновый
  таймер/cron. **Отдельный тикет** (независимый кусок).
- **Системная админка** (пользователи/роли) и её отделение от пользовательских
  настроек — отдельная тема, не здесь.
- Калибровка уровней (rj-tan) и прочие настройки сверх лимитов ретеншна.

## 10. Разобранные находки ревью

- **Решено — хранилище настроек:** заводим в этой фиче, дефолты (10 / 50) — в БД;
  экран редактирования — отдельной задачей (см. §8).
- **Отклонено (MAJOR-1) — удаление/вытеснение партии, пока движок считает ход:** НЕ
  реальный сценарий. Расчёт занимает миллисекунды; авто-вытеснение целит в давно
  брошенные партии (противоположность «сейчас думает»); ручное удаление — со списка,
  где партия не в процессе хода. Спец-обработки не требует; гипотетический осиротевший
  процесс движка добирает штатный idle-sweep.
