# Спека: настройки движка (админ) — конфиг-на-партию, замороженный при создании

**Дата:** 2026-06-16 · **Тип:** feature (бэкенд + фронт).
**Источник истины дизайна:** `docs/superpowers/specs/2026-06-07-renju-design.md`.
**Тикеты:** `rj-8py` (бэк), `rj-h1p` (фронт).

## 1. Проблема и цель

Сейчас уровни сложности (сила/время) живут в `levels.toml`, грузятся в `app.state.levels` один
раз при старте; движковый игрок берёт параметры живьём по `level_id` (`game/players.py`,
`game/deps.py`); NNUE задаётся единым файлом-конфигом движка (`engine/config.toml`), один на все
партии, прошитый в реестр при старте. Калибровать силу можно лишь правкой файла + рестартом; нет
истории, под какими настройками игралась партия.

**Цель:** админ калибрует движок (сила/время уровней + вкл/выкл нейросети) из UI; изменения
применяются к НОВЫМ партиям без рестарта; каждая партия играется под СВОИМ конфигом, и видно,
под каким (история).

## 2. Модель данных

- **Уровень** — стабильная идентичность набора сложности (Новичок … Бог): `id`, имя, порядок.
  **Набор уровней ФИКСИРОВАН** (засевается из текущего `levels.toml`); добавление/удаление уровней
  в этой фиче НЕ предусмотрено — редактируются только значения конфига уровня.
- **Текущие настройки уровней** (то, что правит админ): у каждого уровня — текущие `{strength, timeout}`
  + глобальный `nnue` (один вкл/выкл на все уровни). Хранятся в БД, заменяют файловый `levels.toml` и
  кэш `app.state.levels`.
- **NNUE — глобальный** (один на все уровни), часть конфига. В GET отдаётся один общий `nnue` (по
  определению одинаков для всех уровней). Переключение применяется только к новым партиям.
- **Конфиг партии заморожен при создании.** Партия играется под конфигом `{strength, timeout, nnue}`,
  зафиксированным в момент её создания; он **не меняется** всю жизнь партии (в т.ч. при перезапуске
  процесса движка). Поздние правки админа на идущие/прошлые партии **не влияют** — только на новые.
  Видно, под каким конфигом игралась партия (история).
- **Форма хранения конфига партии — свободный выбор реализации** (решается в плане): ссылка на
  неизменяемую версию-строку конфига / снимок JSON прямо в партии / сборка на лету при запуске
  процесса — равнозначно. Спека фиксирует только СВОЙСТВО (заморожен при создании, не дрейфует,
  история есть), не механизм. Ключевое следствие: параметры партии берутся из её **замороженного
  конфига**, а НЕ из текущего глобального состояния — живой lookup по `level_id` (`players.py`,
  `deps.py`) уходит.

## 3. Поведение

- **Создание партии** на уровне X: фиксируем в партию текущий конфиг уровня X (форма — см. §2/§8).
  Заменяет хранение голого `level_id` + живой lookup.
- **Параметры при игре/восстановлении:** `GameService`/`make_player` берут `EngineParams` из
  **замороженного конфига партии**, а не из глобального словаря уровней. Даже при изменении значений
  админом старая партия видит свой замороженный конфиг.
- **Контракт движка (проверено по исходникам Rapfi):** конфиг движок принимает ТОЛЬКО как **TOML-файл**
  через `--config <path>` (парсер cpptoml; при отсутствии `--config` — встроенный дефолтный TOML). JSON
  и конфиг через stdin он НЕ понимает; по stdin — только команды протокола (`INFO …`). Модель/эвалюатор
  (`nnue`) грузится из TOML при загрузке конфига = при запуске процесса; на живом процессе не меняется.
  → **Чем бы мы ни хранили конфиг в БД, на запуск процесса для партии собирается её TOML-файл и
  передаётся в `--config`.** Встроенный дефолт движка (`InternalConfig`) НЕ используем: он подхватывается
  лишь без `--config` и устарел (не наши настройки), а при переданном `--config` на битый файл движок
  выходит с ошибкой (`main.cpp:113`), не откатываясь на дефолт. Фоллбэк — НАШ закоммиченный базовый
  шаблон (`engine/config.toml`), он валидный конфиг сам по себе; сбой сборки/записи = ошибка спавна,
  её ловит существующий путь восстановления реестра.
- **Применение конфига к процессу движка (база модели «процесс-на-партию»).** Каждая партия поднимает
  свой процесс под СВОЙ конфиг. Механика честно:
  - `strength`/`timeout` шлются командами `INFO` на живой процесс (как сейчас, per-call в `compute_move`).
  - `nnue` определяет, **с КАКИМ TOML-файлом** запущен процесс — выбирается ТОЛЬКО при запуске
    (`--config` в `RapfiProcess.spawn`), на живом процессе сменить нельзя.
  - **Затрагиваемая поверхность реестра** (не «обобщение одного поля»): config-файл партии должен
    доехать до КАЖДОЙ точки спавна. Спавн ленивый: самая ранняя точка — `mark_present` (вход на экран
    партии, `enter`), которая параметров не несёт; плюс спавн в `_claim` из `compute_move`/
    `forbidden_points`; плюс `_respawn` (hang/dead/retry), который сейчас берёт глобальный `self._config`
    (`registry.py:489`). Решение: операции реестра, способные вызвать спавн (`mark_present`,
    `compute_move`, `forbidden_points`), получают файл конфигурации партии; реестр **запоминает его за
    партией** (в её записи `EngineSlot`, рядом с `level_tag`), и `_spawn_into`/`_respawn` берут файл
    оттуда, а не глобальный. Так перезапуск процесса открытой партии не сменит ей нейросеть.
  - **Идущие партии не подхватывают правки:** их замороженный конфиг неизменяем, а процесс при
    перезапуске поднимается под их же файлом конфигурации. Переключение нейросети — только к новым партиям.
- **Эндпоинт `/admin/engine-config` (роль admin, гейт `require_admin` — уже есть):**
  - `GET` — текущие уровни с их текущими конфигами (`strength`/`timeout`) + глобальный `nnue` (берётся
    с любого уровня — инвариант одинакового nnue гарантирует определённость).
  - `PUT` — принимает плоско: таблицу (per-level `strength`/`timeout`) + один глобальный `nnue`.
    Обновляет текущие настройки уровней **атомарно** (одной транзакцией — всё или ничего, иначе
    инвариант одинакового nnue ломается); `nnue` применяется ко всем уровням. (Как именно сохраняется
    новое текущее состояние — новая версия / перезапись текущей строки — зависит от выбранной формы
    хранения, §8.) **Единицы:** хранение/API — миллисекунды (`timeout`); фронт показывает и
    редактирует в секундах и конвертирует с↔мс. Валидация: `strength` 0..100; `timeout` 200..30000 ms
    (нижняя = 0.2 c из мокапа; верхняя согласована с wall-clock-kill `timeout/1000 + 5s`). Возвращает
    обновлённое состояние.
- **Без рестарта:** новые партии читают текущие версии; идущие — на своих (immutable). Рестарт
  приложения/процессов не нужен.

## 4. Фронт

- **Каркас админки** (`/admin`, роль admin): вкладки Пользователи / Движок / Состояние. Каждая
  вкладка — отдельный компонент. «Пользователи» и «Состояние» — заглушка «в будущих релизах» (одна
  строка); их наполнение — отдельные тикеты.
- **Экран «Движок»** (по мокапу `prototype/index.html` → Админ → `#atab-engine`): таблица уровней
  (сила 0–100 + время на ход **в секундах**, как в мокапе; редактируемые поля; имена/набор уровней —
  с бэка, добавления/удаления строк нет) + один глобальный тумблер NNUE + «Сохранить» (PUT). Время
  фронт конвертирует с↔мс (API — в мс, см. §3).

## 5. Что НЕ в этом этапе (вне области)

- **Per-user конфиги** (личные настройки движка у пользователя). СОЗНАТЕЛЬНО отложено — делается
  потом на этой же DB-модели конфигов. Сейчас конфиги только админские, глобальные.
- **Добавление/удаление уровней** — набор фиксирован; меняются только значения.
- **Наполнение вкладок «Пользователи» (`rj-6vk`) и «Состояние»/health (`rj-1in`)** — отдельные
  тикеты; здесь только заглушки в каркасе.
- **Админское управление партиями** — нет в механике.

## 6. Декомпозиция и порядок

Одна фича, одна спека; реализация двумя тикетами:

- **`rj-8py` (бэк):** DB-модель (текущие настройки уровней + заморозка конфига в партию — форма §8),
  Alembic-миграция (+ сид уровней из `levels.toml`, + бэкфилл существующих партий их конфигом),
  параметры партии из её замороженного конфига, конфиг-на-партию в реестре (файл конфигурации
  запоминается за партией, выбор файла под `nnue`, перезапуск процесса — тем же файлом), эндпоинт
  `GET/PUT /admin/engine-config`.
- **`rj-h1p` (фронт):** каркас админки + экран «Движок» на готовом эндпоинте.

Сначала бэк (`rj-8py`), потом фронт (`rj-h1p`).

## 7. Тестирование

- **Домен/DAL:** заморозка конфига партии (правка админа не трогает существующие/идущие партии);
  инвариант одинакового nnue в текущих настройках.
- **Сервис создания партии:** партия получает текущий конфиг уровня; параметры берутся из её
  замороженного конфига (не из глобального словаря).
- **Эндпоинт:** форма `GET`; `PUT` обновляет настройки **атомарно** (частичный сбой → откат всего),
  валидация границ, admin-гейт.
- **Реестр:** процесс партии поднимается под её TOML (по флагу `nnue`) на ВСЕХ точках запуска —
  включая `mark_present` (самая ранняя: вход на экран партии до первого хода); **`_respawn` берёт файл
  СО СЛОТА** (форс-respawn партии с nnue=off → новый процесс снова под off, а не глобальный). Живые
  движковые тесты — по методологии rate-over-N (см. `rj-xv1`), не на конкретный ход.
- **Миграция:** сид уровней/конфигов из `levels.toml`; бэкфилл существующих партий.
- **Фронт:** экран рендерит уровни, `PUT` шлёт правки; каркас/заглушки вкладок.

## 8. Открытые детали реализации (в план, не в спеку)

- **На каждую партию собирается свой TOML и передаётся движку при запуске** (`--config`). Базовый
  шаблон (фиксированные числа: TT, candidate range, тайм-контроль, `coord_conversion_mode=none`, БД,
  `[model] binary_file`) живёт в одном месте; по конфигу партии в него подставляется секция эвалюатора:
  `nnue=on` → `[model.evaluator] type="mix9svq"` + веса; `nnue=off` → секции нет (движок считает на
  классическом эвалюаторе по `binary_file`; классические веса в репо: `Networks/classical/*.bin`).
  Готовый пример `Networks/config-example/gomocalc-classical*.toml` НЕ копируем (у него `coord=X_flipY`
  и пр.) — собираем из нашего шаблона. План **живым прогоном** подтверждает старт off-сборки под
  `rule 4` / `coord=none`.
- **Где живёт собранный TOML партии — ОГРАНИЧЕНИЕ, не просто деталь:** файл должен быть доступен ВСЮ
  жизнь партии (процесс может перезапуститься спустя время, и `_respawn` читает тот же файл) → пишем
  под `data_dir` (`config.py`), а НЕ в эфемерный tmp. Уборка (при выходе/удалении партии) — деталь плана.
- **Форма хранения конфига партии** (ссылка на неизменяемую версию-строку / снимок JSON в партии /
  сборка на лету) — выбор плана. Если выбрана версия-строка: резолв «текущей версии» (монотонная
  `version` на уровень или max-id) + индекс под частый резолв при создании партии.
- **Точные сигнатуры проводки** config-файла в реестре (`mark_present`/`compute_move`/
  `forbidden_points` → слот → `_spawn_into`/`_respawn`) и формат хранения на слоте.
- **Слой сборки `конфиг партии → готовый TOML-файл (с секцией эвалюатора или без)`** — в
  адаптер/конфиг-слое (`app/rapfi`/config), НЕ в домене (домен без I/O). NB: classical example-config
  идёт с `coord_conversion_mode="X_flipY"` — наша off-сборка должна быть на `none`.
- **Миграция существующих партий:** бэкфилл `config_id` на текущую версию их `level_id` (FK NOT NULL
  после бэкфилла) либо nullable FK с fallback — решить в плане.
- **Инвариант «у партии ровно один движковый конфиг»** (актуально, т.к. PvP в дорожной карте).
