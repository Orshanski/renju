# Фронт-срез 1: обёртка + auth — дизайн

- **Дата:** 2026-06-12
- **Тикет:** `rj-0z2` (первый срез эпика `rj-8wf` «Этап 4: фронт/PWA»).
- **Статус:** на ревью.
- **Дополняет:** основную спеку §1/§3/§4.4/§7; auth-фундамент среза 1 этапа 2 (`2026-06-11-user-auth-foundation-design.md`).
- **Бэкенд готов:** auth-эндпоинты `/api/auth/{login,logout,me}` (JWT в httpOnly-cookie, `token_epoch`), CSRF (`X-Requested-With` на небезопасных `/api/`), security-заголовки — всё в `main` (этап 2). Фронт садится на проверенную куку, мокать пользователя не нужно.

## Цель

Первая вертикаль фронта: рабочий **вход + каркас приложения**, на котором дальше растут игровые экраны. Браузер-only. Цель среза — *аутентификация и оболочка работают вживую против реального бэка*, а не контент.

Порядок эпика (брейншторм 2026-06-12, решение Alexey): сперва auth-обёртка (на настоящей куке, без моков), затем игровая доска на хардкод-уровне (срез 2), потом список/новая/восстановление (срез 3) и т.д.

## Скоуп

**Входит:**
- Scaffold `frontend/` (React + Vite + TypeScript).
- Экран логина (username/пароль) + поток `login` → `me` (восстановление сессии) → `logout`.
- `api`-клиент (`fetch`, cookie, CSRF-заголовок, централизованная обработка 401).
- Роутинг (react-router): публичный `/login`, защищённый `/`.
- Auth-состояние (React Context).
- Минимальный shell: header (имя пользователя, «выход») + пустая «домашняя» область-заглушка.
- Раздача собранного SPA бэкендом (FastAPI `StaticFiles` + SPA-fallback) — небольшая бэк-добавка.
- Тест-харнесс фронта (Vitest + RTL + jsdom + msw) и базовые тесты по варианту B.

**Не входит (за scope этого среза — не предлагать как findings):**
- Игровая доска, SSE-клиент, ход/undo (срез 2, `rj-p82`).
- Список партий, новая партия, восстановление по курсору (срез 3, `rj-as6`).
- Экран настроек (срез 4) и бэкенд `/settings`+`user_settings` (`rj-dix`).
- Админ-панель (срез 5) и бэкенд админ-эндпоинтов (`rj-8py`).
- Экран «Правила» (срез 6), PWA-обвязка (`rj-puo`).
- Visual-regression / геометрия-тесты и Playwright (отложены, см. «Тестирование»).

## Сквозные конвенции фронта (устанавливаются здесь, действуют на ВСЕ срезы)

Закреплено брейнштормом 2026-06-12; вынесено в `rj-8wf`.

- **Стек:** React 19 + react-router 7 + Vite + TypeScript. (Стек скопирован с боевого `librarium-py`; расходимся с ним в стилизации и респонсиве — см. ниже.)
- **Стилизация — CSS Modules** (`*.module.css`, scoped). НЕ inline-`style`. CSS Modules — не вкус, а **вынужденный выбор: CSP бэка запрещает inline-стили** (`default-src 'self'` без `'unsafe-inline'`/nonce → браузер блокирует и `<style>`, и React-атрибуты `style={{}}`; именно поэтому librarium-подход на inline тут не взлетел бы — и он же завёл librarium в JS-респонсив). Внешний same-origin CSS под `default-src 'self'` проходит. Токены (цвета, отступы) — один модуль/CSS-переменные.
- **Респонсив — чистый CSS-reflow.** Единый DOM, `@media`, **desktop-first**, брейкпоинт **900px** (`max-width: 900px` → компактная раскладка; `>900` → десктоп). **Без JS-детекта ширины** (нет `useIsMobile`/`matchMedia`-ветвления), **без изолированных mobile/desktop-деревьев**, **без mobile-first**.
  - Первичны: десктоп (браузер на маке) + iPad-ландшафт (~1194/1366) + 12.9"-портрет (~1024). Компакт (≤900): 11"-портрет (~834) + телефон. Телефон ~375 — корнер-кейс, graceful-degrade.
  - Обоснование «чистый CSS, не изолированные деревья»: renju структурно прост (логин/доска/списки/формы) — тяжёлых структурно-разных каркасов нет, поэтому «рендерить оба и прятать» не возникает. Нав на компакте — бургер (CSS прячет ссылки/показывает бургер; открытие — локальный `isOpen`-стейт, это UI-состояние, не детект ширины). Если когда-нибудь всплывёт реально расходящийся каркас — одним швом-хуком в ОДНОМ месте, не render-both-hide.
- **Touch-friendly:** tap-таргеты, ввод не зависит от hover (hover-эффекты — украшение).
- **Тест-дисциплина (вариант B):** строгий TDD на ЛОГИКУ (Vitest + msw для мока `/api`), поведение компонентов — RTL; пиксели/респонсив — **глазами** (Playwright/visual-regression отложены, вернёмся, если глазная приёмка станет болью). jsdom не считает `@media` — поэтому CSS-респонсив не юнит-тестируется by design, проверяется визуально.

## Архитектура и компоненты

### Структура каталогов
Новый top-level `frontend/` рядом с `backend/`:
```
frontend/
  index.html
  vite.config.ts        # dev-прокси /api → FastAPI; build → ../backend/app/static (или dist, см. сборку)
  package.json
  tsconfig.json
  src/
    main.tsx            # точка входа, монтирует <App/>
    App.tsx             # роутер + AuthProvider
    api/client.ts       # fetch-обёртка (cookie, CSRF, 401, JSON, типизованные ошибки)
    auth/
      AuthContext.tsx   # контекст: { user, loading, login(), logout() }
      auth.api.ts       # login/logout/me поверх api/client
      ProtectedRoute.tsx
    pages/
      LoginPage.tsx     # форма входа
      HomePage.tsx      # заглушка «здесь будет список партий» (наполнится в срезе 3)
    components/
      Shell.tsx         # header (имя, выход) + <Outlet/>
    styles/
      theme.module.css  # токены/переменные (цвета, отступы, брейкпоинт-комментарий)
    *.module.css        # рядом с компонентами
```

**Типы и формы ответов (M1, поймано ревью — формы login и me РАЗНЫЕ):**
- Фронт-тип `User = { id: number; username: string; role: string }` (по бэк-`UserDTO`; `role` хранить с самого начала — пригодится для admin-гейта в срезе 5).
- `POST /api/auth/login` → `{ ok: true, user: UserDTO }` — `user` **ВЛОЖЕН**, разворачивать `resp.user`.
- `GET /api/auth/me` → `UserDTO` **плоско** (`{id, username, role}`), брать тело как есть.
- `LoginRequest` бэка: `password` ≤ 72 символа (bcrypt усекает) — форма может мягко ограничить, не критично для среза 1.

### Поток данных (data flow)
- **Старт приложения:** `AuthProvider` при монтировании зовёт `GET /api/auth/me`.
  - 200 → `user` = тело ответа как есть (плоский `UserDTO`), `loading=false`.
  - 401 → `user=null`, `loading=false` (не залогинен; не ошибка).
  - Пока `loading` — рендерим лёгкий сплэш (не редиректим, чтобы не моргать логином при валидной куке).
- **Логин:** `LoginPage` сабмитит форму → `login(username, password)` → `POST /api/auth/login`.
  - 200 → `user` = `resp.user` (развернуть из вложенного), навигация на `/`.
  - 401/ошибка → inline-сообщение «неверные имя или пароль», поле пароля очищается.
- **Logout:** кнопка в `Shell` → `POST /api/auth/logout` → `user=null` → редирект на `/login`.
- **Защита:** `ProtectedRoute` — если `!loading && !user` → `<Navigate to="/login"/>`.

### api-клиент (`api/client.ts`)
- Базовый `request(method, path, body?)`: `fetch` с `credentials: "include"`; на небезопасных методах (POST/PUT/DELETE) шлёт заголовок `X-Requested-With: XMLHttpRequest` (как ждёт CSRF-middleware бэка); `Content-Type: application/json` для тела; парсит JSON.
- **Ошибки типизованы:** `ApiError { status, detail }`. На `401` — централизованно: чистим auth-состояние и редиректим на `/login` (через колбэк/событие, чтобы клиент не знал про роутер напрямую).
- **Исключение — сам запрос логина (N1, поймано ре-ревью):** `POST /api/auth/login` НЕ подпадает под глобальный 401-редирект. Его 401 — это «неверные креды» (результат формы входа), а не «сессия истекла»; `login()` отдаёт ошибку вызывающему `LoginPage`, тот рисует inline. Иначе провал логина дёргал бы и inline-ошибку, и глобальный редирект (редирект-в-себя на `/login`).
- Прод и дев одинаково ходят на относительный `/api/...` (в деве Vite проксирует).

### Роутинг
- react-router: `/login` (публичный, `LoginPage`), `/` (защищённый, `Shell` → `HomePage` в `<Outlet/>`). Неизвестные пути → `/`.

### Состояние
- Только `AuthContext` (`user`, `loading`, `login`, `logout`). Стейт-машин/Redux/Zustand не тащим — в срезе 1 состояние крошечное.

### Бэкенд: раздача SPA
- В `app_factory` после API-роутеров — смонтировать собранный фронт: `StaticFiles` на `/` + **SPA-fallback** (любой не-`/api/`, не-файловый GET → `index.html`), чтобы клиентские роуты выживали на refresh/прямом заходе. Порядок: API-роутеры и middleware — раньше, статика/fallback — последними, чтобы не перехватывать `/api/*`. (В `app_factory` сейчас: middleware refresh→csrf→security_headers, роутеры auth/admin/games, `GET /api/health` — все под `/api/`; статика+fallback добавляются последними.)
- **Граница fallback (явно):** **любой `/api/*`, не пойманный роутером, → 404 JSON от FastAPI, НЕ `index.html`** (иначе фронт получит HTML вместо JSON-ошибки). SPA-fallback на `index.html` (200) — только для НЕ-`/api/` неизвестных путей.
- **Один origin (CSP `connect-src 'self'`):** фронт и API — **тот же origin**; `fetch`/`EventSource` ходят только на относительный `/api/...`. Абсолютные URL к API запрещены (CSP заблокирует). В деве Vite-dev-сервер проксирует `/api`→FastAPI, поэтому тоже один origin с точки зрения браузера.
- **Сборка:** `vite build` → статика в каталог, который читает `StaticFiles` (напр. `backend/app/static/` или `frontend/dist` с путём в конфиге). Точный путь зафиксировать в плане.
- **Дев:** Vite dev-сервер (свой порт) с `server.proxy` `/api` → FastAPI. Фронт и бэк подняты раздельно.

### CSP-совместимость раздаваемого SPA (B1, поймано ревью)
Задеплоенный security-middleware (`backend/app/middleware/security_headers.py`) штампует на **каждый** ответ (включая `index.html`):
`Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'; connect-src 'self'` — **без `'unsafe-inline'`, без nonce/hash**. Следствия для раздаваемого фронта:
- **Никакого inline в `index.html`:** inline `<script>` блокируется `script-src 'self'`, inline `<style>` — `default-src 'self'`. (Сборка Vite по умолчанию способна вставить inline polyfill-`<script>` modulepreload — его убрать.)
- **Никаких inline-стилей в рантайме** (см. конвенцию стилизации): React `style={{}}`-атрибуты CSP блокирует → только CSS Modules / внешний same-origin CSS.
- **Всё same-origin:** JS/CSS/шрифты/картинки — из бандла (внешние `<script type="module" src>`, внешний `.css`); `fetch`/SSE — `connect-src 'self'`.

**Решение среза 1 (минимально-достаточное, БЕЗ правки middleware — строгую CSP сохраняем):** настроить Vite на **CSP-чистую сборку** — `build.modulePreload.polyfill = false` (целевые браузеры — современный Safari на маке/iPad, полифил не нужен), не инлайнить ассеты (`build.assetsInlineLimit = 0`), не добавлять inline `<script>`/`<style>` в `index.html` (в отличие от build-version-скрипта librarium). Альтернатива (nonce/hash в `security_headers.py`) — НЕ берём: сложнее (per-request nonce в `index.html`, чего `StaticFiles` не умеет) и ослабляет CSP. Точные Vite-флаги — в плане.
> **Решение (техническое, безопасность сохраняется):** держим строгую CSP неизменной, сборку делаем CSP-чистой (без inline) — безопаснее и проще, чем ослаблять политику. Основная спека §5.3 допускала nonce/hash для inline Vite как опцию — НЕ берём (слабее, и `StaticFiles` per-request nonce в `index.html` не умеет).

## Обработка ошибок
- `401` на любом `/api` → разлогин + редирект на `/login` (централизованно в api-клиенте).
- Сетевой сбой (`fetch` reject) → пользовательское сообщение, без падения.
- Провал логина: **401** (неверные креды) → inline «неверные имя или пароль»; **429** (rate-limit бэка `auth_service`: >5 попыток / 5 мин с IP) → «слишком много попыток, повторите позже»; прочее → общий «ошибка входа». Не редирект (логин исключён из глобального 401-перехвата, см. api-клиент). Текст 401 не врёт про 429.
- `me` на старте при 401 → трактуем как «не залогинен», тихо (это норма).
- **Ротация куки (refresh-middleware среза 1):** бэк может прислать обновлённый `Set-Cookie` в ЛЮБОМ ответе, в т.ч. на `GET /me` (когда токен старше порога refresh). Фронт ходит `credentials: include` → браузер примет куку прозрачно; читать её фронт не может (httpOnly) — **действий от фронта не требует**, отмечено для полноты.

## Тестирование (вариант B)
- **Vitest + msw (мок `/api`):**
  - `api/client`: шлёт `X-Requested-With` на небезопасных методах, `credentials: include`; парсит JSON; на 401 дёргает разлогин-колбэк; кидает `ApiError` на не-2xx. **Явный кейс: `POST /login` шлёт `X-Requested-With`** (логин — небезопасный метод на `/api/`; без заголовка CSRF-middleware вернёт 403 на самом входе — лёгкая для пропуска ошибка).
  - `AuthContext`: `me` на старте устанавливает `user` (200) / `null` (401); `login` успех/провал; `logout` чистит `user`.
- **React Testing Library:**
  - `LoginPage`: рендер формы, ввод, сабмит зовёт `login`, на провале показывает ошибку и чистит пароль.
  - `Shell`: показывает имя пользователя и кнопку «выход», клик зовёт `logout`.
  - `ProtectedRoute`: при `!user` редиректит на `/login`; при `user` рендерит детей.
- **Респонсив/визуал — глазами** (десктоп + iPad-портрет ~834): один проход приёмки к раскладке; авто-визуал отложен.
- **Смоук (ручной, делает Alexey/я):** `vite build` → поднять бэк со статикой → в браузере: страница рендерится **с пустой консолью (НЕТ CSP-violation)** → логин → shell с именем → refresh (сессия жива) → logout → редирект на логин; проверить защиту: заход на `/` без куки → отдаётся `index.html` (200), затем КЛИЕНТСКИЙ редирект на `/login` (`me`→401→`ProtectedRoute`), а не 401 на самом `/`. Playwright-инфру в этом срезе НЕ ставим (отложена до среза 2, где богаче e2e-потоки).

## Критерии приёмки
1. Сборка фронта отдаётся бэком и **реально рендерится под задеплоенной CSP** (собранный `index.html` без inline `<script>`/`<style>`; консоль без CSP-violation); прямой заход на `/` (с валидной кукой) и refresh не ломают роут (SPA-fallback); `/api/*` мимо роутера → 404 JSON, не index.html.
2. Логин валидными кредами → shell с именем; неверные → inline-ошибка.
3. Refresh после логина сохраняет сессию (через `me`).
4. Logout → редирект на `/login`; защищённый `/` без куки → редирект на `/login`.
5. Раскладка не разваливается на десктопе и iPad-портрете (~834px); компакт ≤900 читаем.
6. `uv`/бэк-тесты зелёные (бэк-добавка статики не ломает существующее); фронт-тесты (Vitest+RTL) зелёные; типы (tsc) и линт фронта чисты.
