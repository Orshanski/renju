# Визуальная консистентность фронта — дизайн (rj-visual)

## Проблема

Экран «пляшет» по горизонтали и вертикали и выглядит разнородно между разделами:
- при переключении вкладок контент прыгает влево/вправо на пару пикселей;
- «Настройки» заметно более сужены/центрированы, чем «Админка»; доска ждёт левее админки;
- разнобой в размерах заголовков между экранами.

**Диагноз (картировано по коду + сверено с прототипом):** это **не плохой дизайн и не редизайн-задача**. Прототип (`prototype/index.html`) — уже консистентный эталон, а React-приложение от него **отъехало** при сборке (каждый экран носит свои ширины/размеры в своём CSS-модуле). Плюс одна чисто-приложенческая болячка, которой в статичном мокапе нет.

Тексты заголовков в приложении **уже совпадают** с мокапом — разъехался только стиль (размеры/ширины/отступы).

## Источник истины

`prototype/index.html` — единственный эталон визуала. Любая правка приводит приложение **к нему**, без переизобретения. Цвета/шрифты/радиусы уже вынесены в `frontend/src/styles/tokens.module.css` (@value-токены); ширины и типошкала — нет, оттого и дрейф.

**Расположение файлов (полные пути):** страничные модули — в `frontend/src/pages/` (`HomePage`, `NewGamePage`, `GamePage`, `SettingsPage`, `LoginPage` — `.tsx` + `.module.css`); админка — в `frontend/src/admin/` (`AdminPage`, `EngineTab`, `UsersTab`); общая оболочка — `frontend/src/components/Shell.{tsx,module.css}`; токены/reset — `frontend/src/styles/`. Ниже в спеке файлы упоминаются краткими именами — каталог по этому списку.

## Четыре причины «бардака» и целевое состояние (всё из прототипа)

### 1. Единая рамка 1120px на всех экранах
Прототип: `.wrap { max-width: 1120px; margin: 0 auto }` (строка 73) — **одна рамка для ВСЕХ экранов**. Заголовки и секции у всех прижаты к одному левому краю.

React разошёлся:
| Экран | `.wrap` сейчас | Цель |
|---|---|---|
| Доска / Новая партия / Игра | 1120px | 1120px (ок) |
| Админка (`AdminPage.module.css`) | **860px** | **1120px** |
| Настройки (`SettingsPage.module.css`) | **680px** | **1120px** |

### 2. Узкий контент — ВНУТРИ рамки, не вместо неё
В прототипе форма/таблицы сужены внутри рамки 1120, прижаты влево:
- настройки: `.settings { max-width: 680px }` (строка 217), без `margin:auto` → влево; `.desc { max-width: 420px }` (строка 220).
- **админка, вкладка «Движок»** (`#atab-engine`): `.sub` `max-width:640px` (строка 468), `.tbl` `max-width:540px` (строка 469), NNUE-`.setrow` `max-width:540px` (строка 481).
- **админка, вкладка «Пользователи»** (`#atab-users`): таблица `<table class="tbl">` в прототипе **БЕЗ** `max-width` (греп по `max-width` её не находит) → намеренно во всю рамку 1120. Сужать НЕ нужно.

React ошибочно схлопнул **всю рамку** до ширины контента (680/860) — оттого настройки «уехали к центру». Цель: рамка 1120 + узкие под-блоки внутри, прижатые влево, как в прототипе.

**ВАЖНО (M1 из ревью):** при расширении `AdminPage .wrap` 860→1120 надо вернуть внутренние `max-width`, которых в React СЕЙЧАС НЕТ — иначе вкладка «Движок» растянется во всю рамку (регресс мимо цели «формы не растянуты»). Сверено по коду: `EngineTab.module.css` — `.sub` (стр.4-10), `.tbl { width:100% }` (стр.12-18), `.setrow` (стр.55-62) — все БЕЗ `max-width`. Цель: вернуть им ограничения по прототипу (sub 640, tbl 540, NNUE-setrow 540). Таблица «Пользователи» (`UsersTab.module.css` `.tbl { width:100% }`, стр.36) остаётся полноширинной — это совпадает с прототипом.

### 3. Единая типографическая шкала
Прототип задаёт **один заголовочный блок для всех экранов** (строки 74-77):
- `.eyebrow` — красный (vermillion) ярлык, UPPERCASE, letter-spacing 5px, **font-size 11px**, с горизонтальным тире слева (`::before { width:26px; height:1px; background: vermillion }`);
- `h1.title` — Shippori Mincho 800, **font-size 40px**, line-height 1.05;
- `.sub` — sumi-soft, **font-size 15px**, weight 300.

Секционные подзаголовки внутри экранов — тоже `.eyebrow` (красный с тире), напр. «Откаты», «Управление партиями», «Сменить пароль» (строки 404/421/439).

React разошёлся, и **глубже, чем размеры — сам стиль eyebrow:**
- **Доска / Новая партия / Игра** — eyebrow сохранили эталон: `color: vermillion`, `fontSerif`, letter-spacing 5px, `::before`-тире (`HomePage.module.css:15-26`, `NewGamePage.module.css:5-6`, `GamePage.module.css:34`). Заголовки 40px. ✓
- **Админка** — eyebrow **разъехался** (`AdminPage.module.css:9-16`): `color: sumiFaint` (серый, не красный), `fontSans`, letter-spacing 2px, **БЕЗ тире** (`::before` нет); `.title` (`:18-24`) — **28px** вместо 40px.
- **Настройки** (`SettingsPage.module.css:10-25`) — то же: eyebrow серый/`fontSans`/2px/без тире; `.title` 28px. Секционные ярлыки — отдельный `.sectionTitle` (тоже серый/sans), а не красный `.eyebrow`.

Именно поэтому «Администрирование» и «Настройки» сейчас НЕ красные и без тире — это и есть просьба Alexey «красным, с тире» = вернуть eyebrow к эталону. Цель: свести eyebrow на ВСЕХ экранах к мокапу (vermillion / fontSerif / letter-spacing 5px / `::before`-тире / 11px), title — 40px, sub — 15px; секционные подзаголовки — тот же красный `.eyebrow` с тире, а не серый `.sectionTitle`.

Шкалу и стиль заголовочного блока вынести в общий слой/токены (рядом с `tokens.module.css`), чтобы не хардкодились повторно по экранам.

**Нюанс вертикальных отступов (m3 из ревью):** общий заголовочный слой унифицирует ТОЛЬКО типографику (font-size/family/weight/letter-spacing/color + тире eyebrow), но **вертикальные `margin` заголовка оставить пер-экранными** — они намеренно разные: `NewGamePage.module.css:7` даёт `.title { margin: 14px 0 22px }` (пробел до плиток уровней) против `6px` снизу у остальных. Если захардкодить общий `6px`, у «Новой партии» подъедут плитки. Аналогично секционные `.eyebrow` в настройках («Откаты»/«Управление партиями»/«Сменить пароль») имеют в прототипе свои инлайн-`margin` (стр. 404 `24px 0 14px`, 421 `30px 0 14px`, 439 `margin-bottom:14px`) — сохранить при замене серого `.sectionTitle` на красный `.eyebrow`, иначе вертикальный ритм формы поедет. Блок «Сменить пароль» — `.passwordBlock { margin-top:30px; max-width:440px }` (`SettingsPage.module.css:182-185`, совпадает с прототипом стр.438): меняем внутри него только `.sectionTitle`→`.eyebrow`, сам `.passwordBlock` (его `max-width`/`margin-top`) НЕ трогаем.

### 4. Резерв места под скроллбар
Под полосу прокрутки место не зарезервировано: `frontend/src/styles/reset.css` (прочитан) задаёт только `* { box-sizing; margin:0; padding:0 }` и `html, body, #root { height:100% }` — никакого `scrollbar-gutter`/`overflow`. На длинном экране скроллбар есть, на коротком исчезает → страница ездит на ~15px при переключении.

NB (поправка Alexey): эта же болячка **есть и в прототипе** — там тоже нет резерва под скроллбар. То есть это не дрейф от мокапа, а общий недочёт обоих; фикс идёт **сверх мокапа** (мокап тоже его не имел).

Цель: `scrollbar-gutter: stable` на корневом скролл-контейнере (`html` в `reset.css`). Горизонтальное дёрганье исчезает.

## Как сделать, чтобы не разъехалось снова (архитектура)

Корень дрейфа — каждый экран носит свои ширины/размеры в своём CSS-модуле. Поэтому:
- **Общая оболочка страницы:** разделяемый слой (компонент-обёртка `Page`/`PageHeader` или общий `layout.module.css`), дающий единую рамку 1120 + заголовочный блок (eyebrow/title/sub). Все экраны используют его, а не копии.
- **Типошкала в токенах:** размеры заголовков — в `tokens.module.css` (или соседнем), а не хардкод по экранам.
- Пер-экранный CSS оставляет только специфику экрана (узкие под-блоки, сетки), не переопределяет рамку/типошкалу.

**Граница общего слоя (уточнение из ре-ревью):** общий заголовочный слой — ТОЛЬКО для **верхнего** блока экрана (eyebrow + title + sub). Внутрисекционные и внутрикарточные eyebrow остаются **локальными** классами своих экранов, их лишь приводим к эталонному стилю (vermillion / fontSerif / 5px / `::before`-тире), НЕ переносим в общий слой:
  - `GamePage.tsx:126` «Лог ходов» (внутри карточки, `GamePage.module.css:46` `.card .eyebrow { margin-bottom:10px }`) — уже эталонный, оставить локальным;
  - секционные ярлыки настроек («Откаты»/«Управление партиями»/«Сменить пароль») — локальные, со своими инлайн-margin (см. §3);
  - админ-вкладки (`EngineTab`/`UsersTab`) своего верхнего eyebrow не имеют — проследить, что общий слой не навяжет им непрошеный стиль.
- **Login в общий слой НЕ заворачиваем** — отдельная fixed-сцена (см. таблицу заголовков).

Это превращает «единую систему» в структурную гарантию, а не дисциплину.

## Что НЕ входит в эту задачу (scope-забор)

- **Это НЕ редизайн.** Не меняем визуальный язык, палитру, шрифты, компоновку экранов — только приводим к уже-одобренному прототипу. (Редизайн был бы отдельной задачей — Alexey выбрал «восстановить консистентность», не «освежить».)
- **Не трогаем логику/поведение/контент.** Игровой процесс, доска, валидация, API, тексты по смыслу — без изменений.
- **Не трогаем адаптив сверх необходимого.** Существующее поведение на `bpCompact` (max-width:900px) сохраняем; правим только то, что мешает консистентности рамки.
- Бэкенд, движок, тесты домена — вне задачи.

- Фронт: `npx tsc --noEmit` 0 ошибок; `npx vitest run` все зелёные. Существующие тесты опираются на роли/тексты, не на имена CSS-классов — но при переносе классов в общий слой убедиться, что селекторы тестов (`getByRole`, `getByText`) не задеты.
- Визуальная приёмка — за Alexey: переключение между всеми экранами без горизонтального/вертикального прыжка; единый левый край заголовков; единый размер заголовков; формы не растянуты во всю рамку.
- **Проверка на узком экране (M2 из ревью):** у админки и настроек сейчас НЕТ своего `@media bpCompact` (горизонтальные отступы даёт общий `Shell .main`). После расширения рамки до 1120 проверить на ≤900px (планшет/телефон), что таблица «Движок» и контент не дают горизонтального переполнения. Внутренние `max-width` (engine tbl/sub 540/640) этому и помогают.
- Эталон для сверки — `prototype/index.html` (открыть `python3 -m http.server --directory prototype`).
