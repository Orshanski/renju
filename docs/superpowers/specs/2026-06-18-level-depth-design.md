# Спека: глубина поиска как параметр уровня

**Дата:** 2026-06-18 · **Тип:** feature (бэкенд + фронт).
**Источник истины дизайна:** `docs/superpowers/specs/2026-06-07-renju-design.md` (§4.5 уровни).
**Тикет:** `rj-23r`. **Стоит на:** `2026-06-16-engine-config-design.md` (настройки уровней в БД, заморозка конфига в партию).

## 1. Проблема и цель

Сила уровня (`strength`) задаёт силу движка **косвенно — через глубину поиска**, но узко: вся шкала силы 0–100 укладывается в глубины 4…16 (замер `backend/scripts/engine_probes/probe_depth_vs_strength_raw.py`), а реально слабый низ (глубины 1–3) силой недостижим — её дно 4. Калибровать тонко нечем.

**Цель:** дать админу второй рычаг — **глубину поиска** (`max_depth`) как явный параметр уровня, рядом с силой. Сила задаёт **потолок** глубины, глубина крутится **под потолком** (в т.ч. в зону 1–3, недостижимую силой). Узлами (`max_node`) НЕ калибруем — отвергнуто (см. §5).

## 2. Модель

- **Зависимость «сила → потолок глубины» зашита в движке** (`SkillMovePicker`, `engine/rapfi/Rapfi/search/skill.h:35-43`): `targetDepth = 4 + ⌊24·(1 − 0.5^(strength/100))⌋`. Переносится в **чистую доменную функцию** `depth_ceiling(strength)` (без I/O, новый модуль в `app/domain/`), покрытую тестами. Развёртка ступеней:

  | Глубина | Сила | | Глубина | Сила |
  |---|---|---|---|---|
  | 4 | 0–6 | | 11 | 50–58 |
  | 5 | 7–12 | | 12 | 59–67 |
  | 6 | 13–19 | | 13 | 68–77 |
  | 7 | 20–26 | | 14 | 78–88 |
  | 8 | 27–33 | | 15 | 89–99 |
  | 9 | 34–41 | | 16 | 100 |
  | 10 | 42–49 | | | |

- **Диапазон глубины уровня:** **верхняя** = `depth_ceiling(своей силы)` (Бог — 99); **нижняя** = **выставленная глубина предыдущего (более слабого) уровня** (НЕ его потолок); у первого уровня (Новичок) нижняя = 1. Глубина крутится `[выставленная глубина предыдущего … свой силовой потолок]`. Зависимость живая: меняешь силу ИЛИ глубину уровня — каскадом сверху вниз пересчитываются нижние границы нижележащих, и выставленное значение каждого снапается в свой новый диапазон. При монотонной силе (зажим §4) диапазоны валидны (нижняя ≤ верх); может выродиться в точку — норма.
- **Бог (сила 100) — особый.** При силе ровно 100 движок выключает skill (`enabled()=level<100`), потолок-от-силы не применяется → сверху глубина силой НЕ ограничена. Поэтому у Бога диапазон **`[16 … 99]`**: **дефолт-старт = 16** (`depth_ceiling(100)`), верхняя = **99** (`max_search_depth`, `engine/config.toml:70`) — куда поднять вручную. Так `max_depth` остаётся обычным `int` (1…99) у всех уровней — никаких «без капа»/`None`/sentinel. NB: дефолт Бога 16 < его текущей фактической силы (skill off давал ~30 по времени) — сознательный калибровочный выбор Alexey (§5), не «идентично».
- **`max_depth` — параметр уровня:** колонка в таблице `levels` (`models/level.py` — `Level`, сейчас `strength`/`timeout_ms`). Набор уровней фиксирован (как в engine-config), редактируются значения.
- **Заморозка в партию.** Замороженный конфиг — `Engine` dataclass (`game/controllers.py`, сейчас `{level_id, strength, timeout_ms, nnue}`) в JSON `game.controllers`. `Engine` += `max_depth`; снимок собирается в `routers/games.py` (`create_game`) из текущего `Level`. `EngineParams` (`domain/engine_params.py`) += `max_depth`; прокидка — в `make_player` (`game/players.py`).

## 3. Поведение

- **Диапазон и подстановка — на ФРОНТЕ, не на бэке (см. §4).** Бэк диапазон НЕ считает и значения НЕ подставляет: принимает и хранит `max_depth` как число, шлёт движку. Вся логика «поле глубины крутится `1…потолок(силы)`, при смене силы переустановить диапазон и, если прежнее значение вышло за верх, поставить верх» живёт во фронте. На бэке диапазонных валидаций/отказов нет.
- **Доставка движку.** `max_depth` шлётся командой `INFO max_depth` в том же per-move блоке `tunable_commands` (`rapfi/protocol.py`), что `strength`/`timeout` — и в `init_commands` (холодный старт), и в `incremental_move_commands` (тёплый ход). `tunable_commands` **ВСЕГДА** включает `INFO max_depth N` (команду не опускаем; `N` — обычный `int` 1…99, у Бога до 99). Так состояние идемпотентно и переживает `_respawn`/persisted-процесс. Анти-инъекция (CLAUDE.md §5.2): в `f"INFO max_depth {N}"` уходит только `int` (гарантия — валидация + frozen `EngineParams`).
- **Эндпоинт `/api/admin/engine-config` (`routers/admin.py`, расширяем существующий, гейт `admin_user`):**
  - `GET` — `LevelConfigDTO` (`dtos/engine_config.py`) += `max_depth` и `depth_ceiling` (потолок; считает бэк — фронт формулу не дублирует); сборка в `_build_engine_config_dto`.
  - `PUT` — `LevelUpdate` (`dtos/engine_config.py`) += `max_depth` (`int`); `ConfigRepository.update` (`config_repository.py`) мутирует `level.max_depth` как есть — без диапазонной логики (её сделал фронт). Атомарно (проверка id до мутаций + `session.commit()` — уже так). На бэке только базовая санитизация типа/границ (`int`, 1…99) для целостности (анти-инъекция), не диапазонная валидация. Нижняя `1` достижима (движок: `min(1,99)=1`), верх `99` = `max_search_depth` — закрепляется raw-тестом (§7).
- **Без рестарта.** Новые партии берут текущий `max_depth` уровня; идущие — свой замороженный.

## 4. Фронт (здесь вся проверка и установка диапазонов)

- **Экран «Движок»** (`/admin` → вкладка Движок, таблица уровней): к строке уровня — **столбец «Глубина»** рядом с «Сила»/«Время».
- **Зажим силы** (держит монотонность уровней): сила уровня ∈ `[сила предыдущего … сила следующего − 1]`; у Новичка нижняя = 1, у Бога верхняя = 100. Монотонность гарантирует ТОЛЬКО фронт; при обходе (прямой PUT с немонотонной силой) диапазоны глубины могут выродиться — бэк это не чинит (вреда нет: self-hosted, admin-only, портит лишь свой движок; CLAUDE.md «не сторожить от несуществующей угрозы»).
- **Диапазон глубины:** `[нижняя … верхняя]` (`depthRange(i, strengths, depths)` в `levelDepth.ts`), где **верхняя** = `depthCeiling(силы)` (Бог — 99), **нижняя** = **выставленная глубина предыдущего уровня** (`depths[i-1]`; у Новичка = 1). Показывается «макс N» (с отступом). При правке силы ИЛИ глубины — `setRow` (`EngineTab.tsx`) делает каскад сверху вниз: зажать силу (`clampStrength`), затем по порядку снапнуть каждую глубину в её `[выставленная предыдущего … верх]`.
  - При равных потолках соседей диапазон — точка (`[4…4]`, глубина фиксирована), норма.
  - Меняешь силу → фронт пересчитывает диапазон; если выставленная глубина вышла за него — ставит **верхнюю границу**.
- Сила и время — без изменений. «Сохранить» (PUT) шлёт строки с числами `strength`/`max_depth`; бэк хранит как есть.

## 5. Что НЕ в этом этапе (вне области)

- **Конкретная калибровка чисел глубины** (какие значения дать уровням). Механизм + UI делаем здесь; шкалу Alexey калибрует сам. По умолчанию сид/бэкфилл = `depth_ceiling(strength)` → для уровней <100 поведение идентично текущему. **Бог (сила 100): дефолт-старт = 16** — сознательно слабее текущего (skill off давал ~30); Alexey так решил, верх 99 поднимается вручную.
- **`max_node` как рычаг — ОТВЕРГНУТ** (не отложен). Замером (`probe baseline`): потребление узлов скачет на порядки по позициям (форс ~50, тихая 530k+ и не сходится) → фиксированный node-cap режет силу непредсказуемо. Глубина предсказуема (горизонт фиксирован). Прочие рычаги Rapfi — вне области.
- **Замена/удаление `strength`** — сила остаётся, это отдельная ось (потолок). **Механику `strength`/`weakness`** не трогаем.
- **Per-user конфиги** — на будущее (как в engine-config).
- `max_depth` в публичную форму партии (`public_view`, `game/controllers.py`) НЕ добавляем — внутренний параметр (как `nnue`).

## 6. Декомпозиция и порядок (бэк → фронт)

- **Бэк:** доменная `depth_ceiling(strength)` (формула из `skill.h`); `EngineParams` (`domain/engine_params.py`) += `max_depth` (дефолт = `99` = `max_search_depth`, чтоб не ломать call-sites; не sentinel — реальный потолок движка); `INFO max_depth` в `tunable_commands` (`rapfi/protocol.py`); `Level` (`models/level.py`) += колонка; `Engine` (`game/controllers.py`) += поле (+ `controller_to_json`/`controller_from_json` с `.get` для недомигрированных); прокидка в `make_player`; снимок в `routers/games.py`; Alembic-миграция (колонка + бэкфилл уровней значением = их потолок; бэкфилл `games.controllers` значением = потолок от **замороженной силы партии** `ctl.strength`, НЕ от текущего `Level` — иначе старая партия задрейфует; `ctl.strength` уже в JSON после миграции `5d790f3dfeb5`); DTO + базовая санитизация (тип, 1…99) в эндпоинте (`dtos/engine_config.py`, `routers/admin.py`, `config_repository.py`) — БЕЗ диапазонной/подстановочной логики (она на фронте).
- **Фронт:** зажим силы соседями + столбец «Глубина» с диапазоном `[выставленная глубина предыдущего … силовой потолок]`, каскадный пересчёт вниз при правке силы/глубины (`levelDepth.ts` `depthRange`/`clampStrength`/`depthCeiling`, `EngineTab.tsx` `setRow`); потолок считается фронтом.

## 7. Тестирование

- **Домен:** `depth_ceiling(strength)` против таблицы §2 (границы ступеней: s6→4, s7→5, …, s90→15, s100→16).
- **Подстановка при смене силы:** сила ушла в другую ступень → глубина = новый потолок; осталась в той же → глубина не трогается; ручное опускание в пределах диапазона сохраняется.
- **Доставка (протокол, pure):** `tunable_commands` включает `INFO max_depth` в cold и warm путях.
- **Рез глубины — raw-прогоном** (НЕ через реестр: `EngineRegistry._read_until` выбрасывает realtime-строки, holds-line судит по геометрии и не различает глубину). `probe_depth_vs_strength_raw.py check`: задал `max_depth` → достигнутая глубина = заданной.
- **Живой движок (реестр):** партия с заданным `max_depth` не ломается (процесс жив, ход легален), по образцу `tests/integration/test_registry_live.py`.
- **Заморозка/сервис:** партия берёт `max_depth` из замороженного конфига; round-trip `controller_to_json`/`controller_from_json` с новым полем.
- **Эндпоинт/миграция:** `GET`/`PUT` с `max_depth`, admin-гейт; миграция сидит уровни = их потолок и бэкфиллит `games.controllers` потолком от **замороженной силы партии** (не текущей) — поведение не меняется для уровней <100. Бог: сид/бэкфилл = **16** (`depth_ceiling(100)`, дефолт-старт; диапазон `[16…99]`, верх 99 крутится вручную) — для Бога сознательно слабее текущего (skill off давал ~30).
- **Фронт:** столбец рендерится, смена силы двигает глубину на потолок, `PUT` шлёт `max_depth`.

## 8. Детали реализации (в план)

- **`max_depth` — обычный `int` 1…99** во всех местах (колонка, замороженный JSON, API DTO); никакого `None`/sentinel. Бог: дефолт-старт/сид = 16 (`depth_ceiling(100)`), верх 99; его НИЖНЯЯ в UI — выставленная глубина Мастера (как у всех уровней, §2). `tunable_commands` всегда шлёт `INFO max_depth N` (§3).
- **Подстановка/диапазон — на фронте** (§4), бэк не участвует. Источник потолка один — бэк `depth_ceiling` (отдаётся в `GET`), фронт формулу не дублирует.
- **Форма хранения в замороженном конфиге** — та же, что в engine-config, +поле.
- **Downgrade миграции** — `drop_column max_depth` (откат JSON-ключа опционален).
