# Renju — дизайн (веб-PWA + Rapfi-бэкенд)

- **Дата:** 2026-06-07
- **Статус:** черновик на вычитку (клиент-сервер, веб, мультиустройство, авторизация, безопасность, bootstrap)
- **Морда:** веб-приложение (PWA) — браузер на маке + установка иконкой на iPad/iPhone
- **Сервер:** Rapfi (движок) + демон-обёртка + хранилище + авторизация/админка + раздача статики
- **Режим:** онлайн-only; self-hosted; партии на сервере, синхронизация между устройствами в реальном времени (SSE)

## 1. Цель и контекст

Своё приложение для игры в **рэндзю** (профессиональные «пять в ряд», 15×15,
несимметричные правила с запрещёнными ходами для чёрных). Self-hosted сервис для
одного-нескольких доверенных пользователей. «Для себя»: играть в браузере и как
PWA на планшете/телефоне, **с любого устройства продолжать ту же партию**.
Соперник от среднего уровня и сильнее. Пользователь играл ~20 лет назад, правила
подзабыл — ценна подсветка запрещённых ходов (её движок отдаёт сам, §4.1).

Ключевое решение: **движок, правила и фолы не пишем сами** — берём готовый
**Rapfi** (чемпион GomoCup) и разворачиваем как сетевой бэкенд. Это снимает самую
сложную часть (рекурсивная детекция фолов) и даёт сильную игру даром.

**Масштаб (честно):** это не «приложение на выходные», а полноценный self-hosted
веб-сервис: морда (PWA), бэкенд (демон + хранилище + авторизация + SSE),
безопасность, админка, развёртывание. Алгоритмической сложности нет нигде
(фолы/ИИ — в Rapfi), но операционного и фронтенд-объёма прилично.

## 2. Не-цели (сознательно режем)

- **Свои фолы и ИИ.** Всё от Rapfi.
- **Нативное iOS-приложение и App Store.** Морда — веб/PWA.
- **Оффлайн-игра.** Движок и состояние на сервере.
- **Сторонние OAuth-провайдеры и открытая саморегистрация** — не на старте.
  Авторизация своя, пользователей заводит админ (§4.4).
- **Встраивание/порт движка на устройство; PvP.** (Дебютные правила в scope:
  позиционные ограничения ходов 1–3 — в MVP, см. §4.10; турнирная балансировка
  свопом и balance-5 — отложена в §12.)

## 3. Архитектура

```
┌───────────────────────┐                    ┌────────────────────────────────────┐
│ Браузер / PWA  (A)    │  POST (ход/действие)  Сервер (self-hosted, один хост)    │
│  доска, ввод, рендер  │ ───────────────────▶ │  reverse-proxy (TLS, заголовки,   │
│  логин, список партий │ ◀──── SSE-поток ──── │   rate-limit) — единств. вход     │
└───────────────────────┘                    │  ┌──────────────────────────────┐  │
┌───────────────────────┐                    │  │ HTTP API + SSE + авторизация │  │
│ Браузер / PWA  (B)    │ ◀──── SSE ───────── │  └──┬────────────┬──────────┬───┘  │
│  др. устройство       │ ───────────────────▶ │  хранилище   stdin/stdout   статика │
│  (тот же пользователь)│                    │  ┌──┴───────┐  ┌──┴──────┐         │
└───────────────────────┘                    │  │ БД       │  │ Rapfi   │ (внутр. │
                                             │  │ users+   │  │ процесс │  сеть)  │
                                             │  │ games    │  └─────────┘         │
                                             │  └──────────┘                       │
                                             └────────────────────────────────────┘
```

Наружу смотрит только nginx (TLS). Rapfi слушает только на localhost хоста, порта
наружу не имеет; **БД — локальный файл SQLite** (не сетевой сервис, наружу нечего
открывать). Браузер → сервер по HTTP (действия), ← по SSE (push). Сервер ↔
Rapfi по протоколу Piskvork. Приложение запускается **одним воркером** (реестр
SSE-подписчиков и отзыв сессий — in-process); дюрабельность SSE даёт лог событий в
БД (reconnect по курсору). N воркеров — на будущее (тогда нужна внешняя шина событий
— Postgres `LISTEN/NOTIFY` или Redis pub-sub — взамен in-process реестра, т.е. и
переезд с SQLite).

## 4. Сервер

### 4.1 Rapfi (движок) — stateless процесс

- **Оценка — NNUE по умолчанию** (секция `[evaluator]` в `config.toml` + веса,
  ~20 МБ для рэндзю, CC0; включается конфигом, без пересборки) — это настоящая сила
  Rapfi. CPU-инференс, GPU не нужен; на 2 ГБ RAM с запасом. **Классика** (паттерн-
  таблица `model220723.bin` ≈ 58 КБ, без `[evaluator]`) — fallback на случай совсем
  слабого CPU.
- **Правило рэндзю:** `RULE 4`. **Фолы даром:** `YXSHOWFORBID` (по уже загруженной
  доске).
- **Сила и настройки — под-ключи команды `INFO`** (не голые команды):
  `INFO STRENGTH n` (0–100, человеческое ослабление — см. §4.5),
  `INFO TIMEOUT_TURN n`, `INFO MAX_DEPTH n`, `INFO MAX_NODE n`,
  `INFO SHOW_DETAIL 2` (включает вывод оценки `INFO EVAL/WINRATE/BESTLINE`),
  `YXNBEST n` (top-N ходов). `STRENGTH` Rapfi не клампит — диапазон валидирует демон.
- **Загрузка позиции:** перед каждым расчётом процесс **переинициализируется**
  (`START 15` → `RULE 4` → нужные `INFO …` → `BOARD … DONE`), иначе состояние
  одной партии протекает в другую. В `BOARD` каждый камень — `x,y,who` (who: 1 = чья
  очередь / 2 = соперник); «чей ход» и цвет человека держит **демон**, не Rapfi.
- **Координатный режим** конфигурируем (`coord_conversion_mode`, default `none`) —
  зафиксировать и согласовать систему координат демон↔клиент.
- **Сборка:** AVX2 не обязателен (скалярный fallback simde); Linux/ARM/Apple
  Silicon; зависимости вендорены. **Провижнинг собирает Rapfi на целевом хосте** под
  его CPU (перенос prebuilt-бинаря между разными CPU → `SIGILL`). Веса — репо
  `dhbloo/rapfi-networks`.
- **Память:** потолок ~350 МБ (настраивается); 2 ГБ — запас.
- Процесс **состояния партии не хранит** — получает полную позицию на каждый расчёт.

### 4.2 Демон-обёртка (наш код)

Мост HTTP/SSE ↔ процесс Rapfi (своего сетевого интерфейса у Rapfi нет).

- «Посчитать ход»: полную позицию → переинициализация (`START 15` → `RULE 4` →
  `INFO …` → `BOARD … DONE`) → ход; `YXSHOWFORBID` → фолы.
- **Парсинг stdout:** Rapfi пишет служебные `MESSAGE/ERROR/DEBUG/INFO` в тот же
  stdout вперемешку с финальной строкой-ходом (голые координаты без префикса). Демон
  выделяет именно строку-ход, `ERROR …` трактует как сбой, `INFO EVAL/BESTLINE`
  собирает при `SHOW_DETAIL`.
- **Один владелец процессов Rapfi** (1 воркер, см. §3): пул/перезапуск,
  **wall-clock kill** зависшего расчёта (таймаут с запасом над `TIMEOUT_TURN`) +
  respawn, лимит одновременных расчётов.
- **Язык:** Python + FastAPI (уже в обиходе; SSE и валидация из коробки).

### 4.3 Хранилище

- **БД:** **SQLite** (один файл на диске хоста) в режиме **WAL**. Профиль — один
  воркер-владелец (см. §3), единицы пользователей, append-лог событий — ровно кейс
  под SQLite; отдельный сетевой сервис БД не нужен. При инициализации движка:
  `PRAGMA journal_mode=WAL` (читатели SSE не блокируют писателя), `foreign_keys=ON`,
  `busy_timeout`. Переход на Postgres (если когда-нибудь понадобится конкурентная
  запись из N воркеров) почти бесплатный — те же модели/миграции, меняется URL.
- **Доступ к БД — только через ORM + репозитории** (SQLAlchemy 2.0 async + aiosqlite;
  Alembic для миграций — родной, с SQLite работает). Сервисы и доменная логика SQL не
  пишут (см. §4.9).
- `users`: `{ id, username, passwordHash, role: admin|user, tokenEpoch, createdAt }`
  (`tokenEpoch` — для отзыва сессий, см. §4.4).
- `games`: `{ id, ownerId, ходы:[[x,y]…], правило, levelId, цветЧеловека, чейХод,
  статус, undoCount, createdAt, updatedAt }` (`undoCount` — для лимита откатов).
  `статус` — единый enum:
  `awaiting_human | engine_thinking | finished_black | finished_white | finished_draw`.
- `sse_events`: append-only лог (автоинкрементный `id`, партия/получатель, payload)
  — дюрабельность SSE и reconnect по курсору (§3, §8).
- `user_settings` (**per-user, глобально**): undo-политика —
  `{ undoEnabled, undoLimit (число | неограниченно), undoAfterGameEnd }`.
  Применяется ко всем партиям пользователя (см. §4.8).
- Источник истины о партии — сервер; клиент держит локальный кэш.

### 4.4 Пользователи и авторизация

- **Роли:** `admin`, `user`. **Вход:** username + пароль (хеш **bcrypt**).
- **Сессия — JWT в httpOnly secure cookie** (`SameSite=Lax` — нужно для PWA;
  `Secure` по env). Нюанс: `EventSource` (SSE) не шлёт кастомные заголовки, но шлёт
  cookie автоматически — поэтому cookie аутентифицирует и `fetch`, и SSE
  единообразно. TTL абсолютный (~7 дней) + sliding refresh при активности.
- **Отзыв сессий — через `token_epoch`** (паттерн librarium): в токене лежит epoch
  пользователя, на каждом запросе сверяется с `users.tokenEpoch` в БД (1 воркер +
  локальная БД → проверка прямо в БД, без in-process кэша). **logout** — удаление cookie
  (epoch не трогаем). **Смена пароля / роли / бан / удаление** — бамп epoch →
  мгновенный отзыв со всех устройств.
- **SSE и отзыв:** при бампе epoch активно рвём открытые стримы пользователя (реестр
  in-process) + стрим перепроверяет epoch на каждом heartbeat — чтобы висящий
  `EventSource` не пережил отзыв сессии.
- **Заведение пользователей:** открытой регистрации нет — создаёт **админ** (§4.7).
- **Защита:** все игровые/пользовательские эндпоинты требуют сессии; партия доступна
  только владельцу (и админу); SSE-подписка проверяет владельца.

### 4.5 Уровни сложности

- Клиент шлёт **enum-идентификатор уровня** (`levelId`, напр. `novice…master`), **не
  сырые числа**. Типобезопасно: TS literal-union на клиенте, Pydantic `Enum` на бэке
  (невалидное = 422).
- **Массив уровней живёт на бэке** — единственный источник правды:
  `id → INFO STRENGTH (+ опц. INFO TIMEOUT_TURN)`. Бэк транслирует `levelId` в
  параметры Rapfi. Клиент получает список уровней (`id` + имя) через `GET /levels`,
  значений STRENGTH не знает.
- `STRENGTH 0–100` — **встроенное человеческое ослабление** Rapfi (режет глубину +
  рандомно выбирает среди топ-кандидатов → настоящие зевки, а не «думает меньше»).
  Мастер = 100 (без рандома) + больше времени.
- Состав массива (сколько уровней, имена, числа) калибруется на реализации.
  **Калибровка чисел — через временный admin-UI** (настройки движка, §4.7): правка
  `STRENGTH`/`TIMEOUT_TURN` по уровням прямо из интерфейса, без захода в код;
  инструмент убирается, когда шкала откалибрована на живой игре. Первый сигнал уже
  есть: `novice` (strength 10) для любителя ощутимо силён — Rapfi с NNUE играет
  крепко даже на низком strength, шкалу, вероятно, двигать вниз.
- Elo-привязки («играй как ~1500») у Rapfi нет — только шкала 0–100; точные рейтинги
  вне MVP.

### 4.6 Контракт API (предварительно)

- **Auth:** `POST /auth/login` (ставит cookie), `POST /auth/logout`, `GET /auth/me`.
- **Levels:** `GET /levels` → список уровней (`id` + имя) для селектора.
- **Users (admin):** `GET/POST /admin/users`, `DELETE /admin/users/{id}`,
  `POST /admin/users/{id}/reset-password`.
- **Settings (per-user):** `GET/PUT /settings` — undo-политика пользователя
  (`undoEnabled`, `undoLimit`, `undoAfterGameEnd`).
- **Games:** `POST /games` `{ levelId }` — **цвет назначается случайно сервером**
  (выбора нет); если человеку выпали белые — сервер сразу инициирует первый ход ИИ.
  `GET /games`; `GET /games/{id}`; `POST /games/{id}/move` — **async**: пишет ход
  игрока, сразу пушит его по SSE, возвращает `202`; ход ИИ считается и приходит
  **отдельным SSE-событием**; `POST /games/{id}/undo` — по undo-политике
  пользователя.
- **SSE:** `GET /games/{id}/events?since=<cursor>` → несёт **любые изменения
  состояния партии** (ход, подсветка запрещённых, статус, откат-усечение, ошибка
  расчёта и т.п. — не закрытый список); reconnect по курсору, `event: reset` если
  курсор протух, heartbeat-пинг, заголовок `X-Accel-Buffering: no`.
- **Admin:** `GET/PUT /admin/engine-config`, `GET /admin/health`.

### 4.7 Админка

Раздел морды под ролью `admin`: пользователи (создать/удалить/сброс пароля/роль),
партии (просмотр/удаление), настройки движка (уровень по умолчанию, NNUE on/off,
лимиты), состояние сервера (жив ли Rapfi, версия, нагрузка).

### 4.8 Состояние партии и «чей ход»

- **Минимальная машина состояний:** `awaiting_human → engine_thinking →
  awaiting_human → … → finished`. Это игровая логика, не concurrency-подсистема.
  `engine_thinking` не застревает: расчёт **идемпотентен** (позиция в БД, Rapfi
  stateless), поэтому при крахе движка / рестарте / повторном обращении к партии он
  просто пересчитывается — отдельный фоновый sweep не нужен (восстановление само
  идёт через durable-БД + SSE-реконнект).
- **«Чей ход» блокирует ввод — главное на фронте:** пока `engine_thinking`, доска не
  принимает клики (индикатор «соперник думает»), мгновенно, без round-trip.
- **Сервер — подстраховка:** отвергает ход «не в свой черёд» (баг клиента /
  отставшее устройство), чтобы в БД не попал мусор.
- **Version-токенов / optimistic concurrency нет** — писатель партии всегда один
  (одиночная игра против ИИ), одновременных ходов с двух устройств не бывает.

**Undo.** Механика: откат **до предыдущего состояния «твой ход»** (обычно убирает
ход ИИ + твой; если партия кончилась твоим ходом — убирает твой последний). Разрешён,
**когда сервер не считает** (не `engine_thinking`) — т.е. твой ход или конец партии
(при `undoAfterGameEnd`); из `finished_*` возвращает партию в `awaiting_human`. Сервер
усекает ходы, пушит по SSE (Rapfi stateless), `undoCount++`. **Политика — глобальная
настройка пользователя** (§4.3 `user_settings`): `undoEnabled` (можно/нельзя),
`undoLimit` (число | неограниченно), `undoAfterGameEnd`. Применяется ко всем партиям;
механика одна, политика определяет «можно / сколько / после конца».

### 4.9 Изоляция слоёв (строго)

Чёткое разделение ответственности; каждый слой тестируется отдельно:

- **Роутеры (API)** — тонкие: HTTP-обвязка (валидация запроса, вызов сервиса,
  формат ответа). Логики нет.
- **Сервисы** — оркестрация use-case'ов (создать партию, сделать ход, undo, завести
  пользователя). Координируют; в БД сами не лезут и доменные правила не считают.
- **Домен** — чистая логика: машина состояний партии, undo-политика, маппинг
  уровня→`STRENGTH`, статусы. Без I/O — легко тестируется.
- **Репозитории / DAL** — единственный слой, что трогает БД (через ORM). Сервисы
  зовут репозитории, не пишут SQL.
- **Адаптер Rapfi** — отдельный модуль (процесс, протокол, парсинг) как внешний
  сервис; домен и сервисы от деталей протокола изолированы.
- **SSE-хаб** — инфраструктурный компонент (рядом с адаптером Rapfi): владеет
  in-process реестром подписок, рассылает события, умеет `revoke(userId)` (teardown
  стримов при бампе epoch, §4.4). Сервисы публикуют события через него.
- **Фронт** — аналогично разделён: компоненты (UI) / api-клиент (сеть) / состояние;
  рендер доски изолирован в своём модуле.

### 4.10 Дебютные правила рэндзю (RIF)

Профессиональное рэндзю определяется не только фолами, но и **дебютными
правилами**: без них преимущество чёрных делает игру нечестной. Полную процедуру
RIF (Renju International Federation) фиксируем здесь целиком, чтобы не потерять; в
MVP реализуем только позиционную часть (**Уровень 1**), балансировку свопом —
позже (§12). Источники: `renju.net/starting`, `renju.se/rif/rifrules.htm`, RIF (§13).

**Полная процедура RIF:**

| Ход | Сторона | Требование |
|-----|---------|------------|
| 1 | чёрные | центр доски — H8 = (7,7) |
| 2 | белые | внутри центрального **3×3** (2 неэквивалентных по симметрии варианта) |
| 3 | чёрные | внутри центрального **5×5** (13 вариантов → 26 классических дебютов: 13 прямых + 13 непрямых) |
| своп | — | после 3-го хода «временный белый» выбирает, кто играет чёрными/белыми (право поменяться сторонами) |
| 4 | белые | любая свободная клетка (без ограничений) |
| 5 | чёрные → белые | чёрные предлагают **два разных** варианта 5-го камня; белые выбирают один как 5-й ход партии (balance-5) |
| 6+ | — | обычная игра с фолами (§4.1) |

Позиционные ограничения 1–3 **сужают** дебют к центру (а не отодвигают от него) —
чтобы он попал в один из 26 классифицированных паттернов. Уравнивают преимущество
чёрных не квадраты, а **своп цвета + balance-5**.

**В MVP — Уровень 1 (позиционные ограничения ходов 1–3):**

- **Ход человека:** доменная валидация дебютного ограничения (чистая функция,
  слой §4.9) + блок ввода на фронте (механика как у фолов §7): в недопустимые на
  дебюте клетки просто не даём ставить, плюс подсветка разрешённого квадрата.
- **Ход ИИ:** Rapfi RIF-дебют не соблюдает (`RULE 4` маппится в `{RENJU,
  FREEOPEN}` — проверено в `command/gomocup.cpp`; движок-чёрный сам ходит внутри
  квадрата нелегально). Ограничиваем его командой **`YXBLOCK`** (`gomocup.cpp`):
  она читает клетки `x,y` до `DONE` в список `blockMoves`, который поиск исключает
  из корневых ходов. Перед запросом дебютного хода ИИ блокируем все клетки **вне**
  нужного квадрата → движок выбирает сильнейший ход **внутри** (проверено живыми
  прогонами: 2-й ход → внутри 3×3, 3-й → внутри 5×5). `YXBLOCKRESET` сбрасывает
  блокировку после дебюта. 1-й ход чёрных ставим в центр сами (или `YXBLOCK` всего,
  кроме центра).
- Цвет назначается случайно (§4.6); балансировки свопом в Уровне 1 нет — для игры
  человек-против-ИИ баланс регулируется **уровнем сложности** (§4.5), не свопом.

**Реализация Уровня 1 (модули):**

- `app/domain/opening.py` — чистая функция
  `opening_zone(move_count) -> frozenset[Point] | None`: ход 0 → `{(7,7)}` (центр),
  ход 1 → центральный 3×3, ход 2 → центральный 5×5, ход ≥3 → `None` (без
  ограничений). Единственный источник правды о дебютной зоне; им пользуются и
  валидация, и адаптер, и UI.
- **Серверная валидация (защита, не UX):** `validate_human_move`
  (`app/domain/game.py`) получает доп. проверку — если `opening_zone(len(moves))`
  не `None` и ход вне зоны → `MoveRejected` с новой причиной `OPENING_VIOLATION`
  (в `MoveRejectReason`). Отвергается **всегда**, независимо от фронта (§4.8/§5.2);
  фронт-блок — необязательное зеркало.
- **Обуздание ИИ (без протечки домена):** `RapfiAdapter.compute_move` принимает
  опциональный `allowed_zone: frozenset[Point] | None`; при заданной зоне — `YXBLOCK`
  всех свободных клеток **вне** неё перед запросом, `YXBLOCKRESET` после. Адаптер про
  «дебют» не знает — изоляция §4.9 цела. 1-й ход ИИ-чёрного = `allowed_zone={(7,7)}`.
- Подсветка зоны (фронт / play_cli) — зеркало `opening_zone`, чисто UX.

**Позже (§12) — Уровни 2–3 (балансировка):** своп цвета после 3-го хода (пересмотрит
«цвет назначается случайно», §4.6) и balance-5 (два предложения + выбор). Движок
несёт нужные примитивы: `YXNBEST` (top-N ходов), `YXBALANCEONE`/`YXBALANCETWO`
(balance-команды), а также собственные swap-механики (`OpeningRule SWAP1/SWAP2`,
`ActionType::Swap`/`Swap2PutTwo` в `core/types.h`). Но движковый swap привязан к
freestyle (`RULE 5/6`) и процедурно это Gomocup-swap2, **не** RIF-своп — для RIF
понадобится наша надстройка (выбор делает игрок, движок только оценивает).

## 5. Безопасность

Закладывается в дизайн. Сервис self-hosted и смотрит в интернет.

### 5.1 Сетевая защита

- **TLS/HTTPS** везде (reverse-proxy, авто-TLS), редирект 80→443, **HSTS**.
- **Наружу открыт только nginx** (443/80). Rapfi слушает только на
  localhost хоста; БД — локальный файл SQLite (сети не слушает).
- **Rate limiting:** жёстко на `/auth/login` (антибрутфорс) и на игровые/API-
  эндпоинты (антифлуд, защита ресурсов Rapfi) — на reverse-proxy и/или в приложении.
- **Антибрутфорс логина:** лимит по **IP** + нарастающая задержка (блокировку
  *аккаунта* избегаем — это вектор DoS на чужой аккаунт).
- **CORS:** строго собственный origin; API не для сторонних сайтов.
- **Secure-заголовки:** HSTS, `X-Content-Type-Options: nosniff`, `Referrer-Policy`,
  CSP (см. 5.3).
- **Дорогие расчёты — невозможны by design:** клиент не задаёт сырые параметры
  движка, только enum-уровень из server-side списка (§4.5), бэк транслирует в
  ограниченные значения. Отдельная анти-DoS-очередь не нужна — анонимов нет.

### 5.2 Защита от инъекций

- **SQL:** только параметризованные запросы / ORM; конкатенация SQL со вводом
  запрещена.
- **Инъекция в протокол Rapfi (критично для нашей архитектуры):** ходы уходят в
  процесс Rapfi по текстовому stdin-протоколу. Сырой пользовательский ввод в stdin
  **никогда** не попадает. Ход строго валидируется: ровно два целых в `0..14`; всё
  иное (переводы строк, лишние токены, команды протокола) отвергается **до**
  передачи движку. Так клиент не «допишет» команду Piskvork.
- **Валидация всего API-ввода** по схемам (Pydantic и т.п.): типы, диапазоны,
  отказ неожиданных полей.
- **XSS:** пользовательский контент (имена и т.п.) экранируется при рендере; без
  `innerHTML` с непроверенными данными; CSP как второй рубеж.

### 5.3 Защита от скрытой переадресации и смежного

- **Open redirect:** у нас SPA с клиентским роутингом — серверный redirect-параметр
  после логина **не используем** (нет вектора). Появится — whitelist относительных
  путей.
- **Clickjacking:** `X-Frame-Options: DENY` + CSP `frame-ancestors 'none'`.
- **CSP — полный набор:** `default-src 'self'`, `script-src 'self'` (+ nonce/hash для
  inline Vite, без `'unsafe-inline'`), `object-src 'none'`, `base-uri 'none'`,
  `frame-ancestors 'none'`, **`connect-src 'self'`** (под SSE); согласовать с service
  worker PWA.
- **CSRF** (следствие cookie-сессий): `SameSite=Lax` + проверка `X-Requested-With`
  (double-submit) на мутирующих запросах.
- **Cookie-флаги:** `httpOnly`, `Secure` (по env), `SameSite=Lax` (Strict ломает вход
  в PWA по внешней ссылке).
- **Лимиты:** размер тела запроса и длина `ходы[]` (≤ 225) ограничены.
- На будущее: при появлении OAuth-провайдеров — строгая валидация `redirect_uri`.

## 6. Первый старт и развёртывание (bootstrap, day-0)

То, что чаще всего забывают; должно работать из коробки при первом подъёме.

**Dev — нативно на маке (Apple Silicon), без VM и без Docker.** Весь стек идёт
локально: **Rapfi собирается на маке** (Apple clang, `arm64-NEON-DOTPROD` — собран и
проверен, ходит + отдаёт фолы), бэк (**FastAPI** под `uvicorn`, один воркер) и
**SQLite** кроссплатформенны, фронт — `vite`. Запуск на `localhost`. ВМ для
разработки не нужна; то, что движок одинаково собирается и под Linux/ARM (отдельно
проверено), и под Apple Silicon, снимает прежнюю причину для dev-VM.

**Prod — Linux-сервер (VPS), без Docker** (поднимать весь стек в контейнерах — лишний
overhead). Сервисы ставятся нативно: **nginx** (reverse-proxy + TLS), **FastAPI** под
`uvicorn` (один воркер) через **systemd**, процесс **Rapfi** (под управлением
приложения или systemd-юнитом, **пересобирается под CPU прода** — prebuilt-бинарь
между разными CPU не переносить). БД отдельным сервисом не ставится — это **файл
SQLite** на диске сервера.

- **Воспроизводимость прод-развёртывания:** provisioning-скрипт (bash/Ansible) +
  systemd-юниты для prod-хоста — замена «единого Docker-образа», против дрейфа «на
  сервере работало». Для dev разворачивать нечего (всё локально на маке). Подготовку
  прод-хоста (тулчейн + сборка Rapfi под его CPU) удобнее держать **инструкцией-
  runbook**, а не скриптом, — операция разовая, на одной машине.
- **Миграции:** версионируемые **Alembic**, применяются **pre-start** (отдельный
  шаг / `ExecStartPre`), не «при первом запросе» (иначе гонка/неопределённость).
- **Создание первого админа:** CLI-скрипт `create_admin` (паттерн librarium) —
  создаёт админа, если их ещё нет; повторный запуск существующих не трогает. Просто и
  достаточно для self-hosted (env/setup-экран не усложняем).
- **Сидинг** дефолтной конфигурации движка. **Идемпотентность:** повторный
  провижнинг/перезапуск не ломает данные.
- **Сеть:** наружу только nginx (443/80, редирект 80→443); Rapfi — на
  `localhost` хоста, БД — локальный файл (сети не слушает). **HTTPS обязателен** (PWA
  service worker + secure cookie + стабильный SSE по HTTP/2) — на проде **nginx +
  certbot/Let's Encrypt** (Caddy с авто-TLS не используем, выбран nginx). **На dev**
  — **мак в Tailscale**: `tailscale serve` терминирует валидный HTTPS (MagicDNS
  `*.ts.net`) на локальный `uvicorn`, без nginx/certbot и самоподписанных сертов.
  Бонус: реальные iPhone/iPad в той же tailnet тестируют PWA по валидному HTTPS, не
  публикуя сервис в интернет. (Ставится, когда появится фронт для тестов с устройств.)
- **Секреты** (session secret, bootstrap-пароль, креды БД) — через env /
  systemd `EnvironmentFile`, не в репозитории.
- **Бэкап SQLite:** горячая копия через `sqlite3 .backup` / `VACUUM INTO` по cron
  (не `cp` по живому WAL-файлу), копия **off-host** (VPS может умереть целиком),
  **протестированное** восстановление, retention; бэкап **до** миграции при апгрейде.
  Опционально — `litestream` для непрерывной репликации в S3-совместимое хранилище.
  (Паттерн librarium: dump + sync во внешнее хранилище — тот же, только дамп проще.)
- **Логи/мониторинг/аудит:** structured-логи + ротация (journald); **audit-trail**
  auth/admin-операций (логин, создание/удаление юзера, reset пароля, смена роли);
  алерт на падение через systemd `OnFailure`; `GET /admin/health`.

## 7. Веб-морда (PWA)

Морда **тонкая по логике** (правил игры не знает — считает сервер), но **по объёму
самая большая часть проекта**: рендер, ввод, состояние, синхронизация, экраны
(логин, список партий, игра, админка), стили, PWA-обвязка. «Тонкий клиент» — про
ответственность, не про трудозатраты.

- **Экран логина**; для admin — доступ к админке.
- **Доска 15×15** на Canvas/SVG: сетка, камни, hover-превью, клик/тап.
- **Список партий / продолжить** с любого устройства; новая партия — выбор уровня
  (цвет назначается случайно, без выбора).
- **Экран настроек пользователя:** undo-политика (вкл/выкл, лимит, после конца).
- **Стек:** **React + Vite + TypeScript**; сборка `vite build`, статику отдаёт
  бэкенд (FastAPI `StaticFiles`). Действия — `fetch` (cookie); обновления —
  `EventSource` (SSE) с авто-reconnect. Доску рисуем на Canvas/SVG внутри
  компонента — React несёт обвязку (экраны, состояние, реактивность к SSE), не доску.
- **PWA:** через `vite-plugin-pwa` (генерирует service worker + манифест) →
  установка иконкой, полноэкранно, кэш оболочки.
- **Админ-экраны** (роль admin).
- **Экран «Правила»:** справка по рэндзю — камни на пересечениях, цель (ровно пять),
  запрещённые ходы чёрных (3×3 / 4×4 / оверлайн) с пояснениями, асимметрия (белым
  оверлайн = победа). Статический обучающий контент по тексту RIF (§13).
- **Локальный кэш** партии; сервер — источник истины. **Undo**, **селектор уровня**
  (по `id`, см. §4.5).
- **Подсветка запрещённых точек = блокировка ввода:** в них просто **не даём
  ставить** (как в занятую клетку) — фол невозможен, это не исход партии. Актуально,
  когда человек играет чёрными (у белых фолов нет, ИИ их не делает).
  Опционально — **всплывающая подсказка** при наведении («чёрным здесь нельзя»).
  Точный тип (двойная тройка/четвёрка/оверлайн) Rapfi наружу **не отдаёт**
  (`YXSHOWFORBID` — только координаты, проверено), поэтому подсказка **общей
  формулировкой**; точный ярлык потребовал бы своей классификации линий (вне цели).
  Подробности типов — на экране «Правила». Обучающая фича.
- **Рендер канвы — бесшовный:** свой ход рисуем **оптимистично** сразу по клику; пока
  `engine_thinking` — ввод заблокирован + индикатор «соперник думает»; ход ИИ приходит
  SSE-дельтой и **дорисовывается одним камнем с анимацией** (сетка — статичный слой,
  доску целиком не перерисовываем).
- **Оптимистичный ход — с откатом:** свой камень держим как **неподтверждённый**;
  сервер принял → закрепляем; сеть упала / сервер отверг (ход не в свой черёд,
  рассинхрон) → камень **убираем**, ресинхрон с сервером (он — источник истины) +
  понятное сообщение / повтор. Молча ход не теряется.
- **SSE:** `EventSource` с курсором (`?since`, курсор в localStorage), авто-reconnect
  и до-синхрон после разрыва.

## 8. Игровой цикл (серверное состояние + SSE)

1. Игрок кликает (если его ход) → клиент рисует камень **оптимистично** →
   `POST /games/{id}/move` → `202`. Статус → `engine_thinking`, ввод на фронте
   заблокирован, индикатор «думает».
2. Сервер пишет ход игрока, сразу пушит его по SSE; затем полную позицию демону →
   Rapfi → пишет ход ИИ, `forbidden`, статус.
3. Сервер **пушит по SSE** всем устройствам: ход ИИ, `forbidden`, `status`. Канва
   дорисовывает камень анимацией.
4. Все устройства применяют события (дельты) и синхронны.
5. **Undo:** `POST /games/{id}/undo` → откат пары ходов (по undo-политике
   пользователя: можно / лимит / после конца) → усечение → push.
6. **Цвет:** назначается **случайно** при создании; выпали белые человеку → сервер
   сразу делает первый ход ИИ.

## 9. Хранение и синхронизация

- Источник истины — сервер (БД). Партия доступна с любого устройства, изменения
  раздаются по SSE live.
- Клиент кэширует партию локально для быстрого старта и переживания разрывов; при
  расхождении выигрывает сервер. Реплей — даром.

## 10. Ошибки и краевые случаи

- **401/403** — нет сессии / чужая партия / не-admin → редирект на логин или отказ.
- **Сеть/таймаут** — сообщение и повтор; кэш не теряет отображение.
- **SSE-разрыв** — авто-reconnect по курсору; `event: reset` если курсор протух →
  до-синхрон через `GET /games/{id}`. При реконнекте — проверка сессии (`/auth/me`):
  401 → на логин (иначе после отзыва сессии клиент бесконечно ретраит).
- **Крах/таймаут Rapfi** — демон перезапускает процесс (wall-clock kill зависшего) и
  **пересчитывает идемпотентно** (позиция в БД), пушит по SSE; партия не зависает.
  Застрявшие в `engine_thinking` досчитываются при обращении к партии (см. §4.8).
- **Ход не в свой черёд** — фронт не даёт ходить вне `awaiting_human`; сервер
  отвергает как подстраховку (version-токенов нет — писатель один).
- **Оптимистичный ход не прошёл** (сеть / отказ сервера) — фронт откатывает
  неподтверждённый камень и ресинхронизируется с сервером (истина); можно повторить.
- **Конец игры** — по `status`: пятёрка, оверлайн белых, **ничья** (полную доску
  фиксирует демон по числу камней).
- **Фол** — не исход: запрещённый ход блокируется на вводе (фронт + серверная
  подстраховка), как занятая клетка.

## 11. Тестирование

- **Авторизация/роли:** логин, защита эндпоинтов, владелец vs чужой, admin vs user;
  **отзыв сессий через `token_epoch`** (logout / смена пароля / удаление), включая
  закрытие висящих SSE-стримов.
- **Безопасность:** rate-limit/брутфорс логина; отказ инъекций — **особенно
  protocol-injection** (ход с `\n`/лишними токенами не должен пройти в Rapfi);
  SQL-инъекция; open-redirect отвергается; наличие secure-заголовков и CSRF-защиты.
- **Bootstrap:** первый старт с пустой БД (создание админа), идемпотентность,
  миграции.
- **Демон:** интеграция против реального Rapfi (позиция → ход и фолы; падение);
  **парсинг stdout** (игнор `MESSAGE/ERROR/INFO`, выделение строки-хода),
  переинициализация, wall-clock kill зависшего.
- **Хранилище/API:** CRUD партий и пользователей, контракт.
- **SSE:** доставка, реконнект, ресинхронизация.
- **Корректность фолов — забота Rapfi**, сами не тестируем.
- **Морда:** применение событий, undo, отрисовка, кэш, экраны.

## 12. Открытые вопросы и будущее

- **Claude-тренер** — слой на сервере (вызовы API наружу).
- **Библиотека обучающих партий с комментариями** — аннотированная коллекция в
  пару к Claude-тренеру (разбор и обучение). Не в MVP.
- **Открытая регистрация / инвайты / OAuth-провайдеры** (+ валидация redirect_uri).
- **Балансировка дебюта (Уровни 2–3, см. §4.10):** своп цвета после 3-го хода
  (пересматривает «цвет назначается случайно», §4.6) и balance-5 (два предложения
  чёрных + выбор белых). Позиционные ограничения ходов 1–3 — уже в MVP (§4.10).
- **PvP (человек против человека)** — архитектура предрасположена: серверные партии +
  SSE-реалтайм + мультиустройство уже есть. Добавить: модель двух игроков-людей,
  приглашение/присоединение (ссылка/код-комната), presence + обработку отключений,
  маршрутизацию ходов + серверную проверку «чей ход» (уже есть). **Rapfi переходит из
  игрока в арбитра** — проверяет фолы/победу/подсветку (уже умеет), ходов не делает.
  Надстройка, не переписывание. Вне MVP.
- **N воркеров** — внешняя шина событий вместо in-process реестра (Postgres
  `LISTEN/NOTIFY` или Redis pub-sub); потянет за собой и переезд с SQLite на Postgres.
  Если когда-нибудь понадобится масштаб.
- Финальный контракт API и формат SSE-событий — на реализации (рабочую реализацию
  SSE свериться с `librarium-py` — тот же Python-стек).
