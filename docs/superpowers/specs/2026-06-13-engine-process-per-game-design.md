# Спека: процесс Rapfi на игру (rj-899)

**Статус:** draft (каркас одобрен Alexey 2026-06-13) · **Тикет:** rj-899 (P2, feature) · **Дата:** 2026-06-13
**Связанные:** rj-t95 (движковый форк-фикс отката — отдельный трек), rj-as6 (фронт-срез 3), rj-h0y (выход на главный)

## 1. Проблема

Сервер держит **один** долгоживущий процесс Rapfi (`app.state.adapter: RapfiAdapter`,
один `self._proc`, единый `asyncio.Lock`). На каждый запрос переинициализируется только
позиция (`START`/`INFO`/`BOARD`-реплей); процесс живёт от старта сервера до смерти. Это
неверно by design:

- Один процесс держит **одно** инкрементальное состояние под **одну** партию: NNUE-
  аккумулятор (`Evaluator::accumulator[BLACK|WHITE]`) и transposition table.
  Мультиплексировать игры можно лишь при безупречном сбросе всего состояния на каждый
  запрос — хрупко (доказано в **rj-t95**: при укорочении позиции/откате аккумулятор
  `mix9svq` протухает и персистит; `START` его не чистит) и приковывает к `Lock=1`.
- Полный `BOARD`-реплей каждый ход — техдолг общего процесса.

## 2. Цель

**Своя OS-инстанция Rapfi на каждую партию.** Партии изолированы физически; параллельны
(общий `Lock` снят). **Наблюдаемость:** при игре на нескольких устройствах из логов
видно, где какой процесс — ключ `game_id` (он же в URL `/game/:id`) + OS `pid`.

## 3. Дизайн

### 3.1 Реестр процессов — изолированная подсистема (app.rapfi)

`RapfiAdapter` → **`EngineRegistry`** (`app.rapfi`): самостоятельный сервис, владеющий
процессами по `game_id`. enter/leave/SSE — **сигналы, втекающие в него**; реестр про
HTTP-эндпоинты и SSE **не знает** (зависимость router → реестр, не наоборот).

- `dict[game_id → EngineSlot]`. `EngineSlot` = `proc` + `pid` + `io_lock` (сериализует
  обмен с процессом партии) + `inflight: int` (идущие расчёты — защита от kill во время
  расчёта) + `presence: int` (сколько устройств держат партию открытой) + `last_activity`
  + `level_tag`.
- **`registry_lock`** (короткий) защищает `dict` и поля-счётчики слота. НЕ держится во
  время spawn (загрузка весов) и расчёта.
- **Один процесс на `game_id`, check-before-spawn:** вход со 2-го устройства находит
  существующий слот и **переиспользует** его (get-or-create под `registry_lock`,
  атомарно, с placeholder+Event для дедупа конкурентных обращений; провал spawn чистит
  placeholder).
- **Слойность:** реестр оперирует только `game_id` + `EngineParams` + opaque `level_tag:
  str` (для логов); `app.game`/`level_id` не импортирует. `app.state.advancing` остаётся
  (независимый дедуп `advance`).
- `RapfiProcess` получает свойство `pid`; `protocol.py` по протоколу не меняется.
- Текущие гарантии `RapfiAdapter` (retry-once на свежем процессе, wall-clock kill+respawn
  при hang/death) **переезжают per-slot**: завис/умер процесс `game_id` → respawn ЭТОГО
  слота + retry-once (инвариант «один процесс на `game_id`» сохраняется; respawn меняет
  `pid` — это норма, логируется `reason=hang`). Глобальный `adapter._lock`/`_proc`
  растворяются в реестре (логика `_request`/`_attempt` переезжает в слот).

### 3.2 Присутствие и жизненный цикл

**Присутствие — счётчик `presence` на `game_id`** (сколько устройств в партии):

- **`enter` (presence++) — автоматически при входе в партию** (открытие игрового экрана;
  enter летит **с экрана**, не из `create_game` — см. §3.6). Гарантированно срабатывает —
  иначе изолированный реестр не получит сигнал
  и процесс не поднимется. **Энергичный спаун:** на `enter` процесс поднимается сразу
  (а не лениво на первый ход). 0→1 → spawn; presence>0 уже есть → переиспользуем (check).
- **`leave` (presence--) — явный эндпоинт** при выходе на главный экран. 1→0 (ушло
  последнее устройство) → **release** процесса. Один `leave` при нескольких устройствах
  процесс **не трогает** (presence остаётся >0).

**Два триггера гашения** (+ shutdown как housekeeping):

1. **Выход** — `leave` довёл `presence` до 0 **и `inflight==0`** → release сразу. Если
   `inflight>0` (идёт расчёт) — release НЕ делаем и отложенный флаг НЕ ставим (продьюсера
   отложенного релиза нет): `presence` остаётся 0, слот доберёт idle-sweep по простою.
2. **Таймаут простоя (idle-sweep)** — периодический фоновый таск реапит слоты с
   `inflight==0 && now-last_activity > idle_timeout` (по **неактивности**, БЕЗ гейта по
   `presence`). `last_activity` обновляется на **любом** обращении к слоту (enter, ход
   человека, `compute_move`, `forbidden_points`) — активная партия держится тёплой сама,
   idle её не берёт; брошенная (в т.ч. с зависшим `presence>0`: устройство умерло без
   `leave`) реапится по простою. `inflight==0` страхует от убийства идущего расчёта
   фоновой партии (где `presence` и так 0). (Роли независимы: `presence` — устройства,
   гейтит триггер ВЫХОД; `inflight` — идущие расчёты, защита от kill-during-compute.)
3. **shutdown** — гасим все слоты (housekeeping, не лайфсайкл-триггер).

**Расчёт изолирован от входа/выхода (compute-safety):** `leave`, доведший `presence` до
0, **НЕ прерывает идущий ход** движка. Защита — `inflight`: инкрементится при выборе
слота **в той же секции `registry_lock`** (TOCTOU невозможен), release/sweep гасят только
`inflight==0`. Процесс с идущим расчётом дорабатывает ход; освобождение — не немедленное, его доберёт
idle-sweep по простою. Движок может посчитать ход в фоне (`advance`) независимо от
presence: `compute_move`/`forbidden_points` сами поднимают процесс по `game_id`
(get-or-create), даже при `presence==0` (фоновая партия без зрителя); такой процесс
уберёт idle (а enter с экрана, если откроют, переиспользует его).

**Undo не трогает слот** — within-game загрязнение после отката это rj-t95; слот
переживает undo/реконнект. (Откат после окончания партии разрешён, `UndoPolicy.
after_game_end` — ещё одна причина не привязывать процесс к статусу партии.)

**Завершение партии процесс НЕ гасит** — финиш обратим откатом; держим тёплым, уберёт
idle/leave.

### 3.3 Без лимита процессов (пока)

Потолка/LRU/насыщения **нет** — на сервере 3–5 игроков, потолок не нужен (явное решение
Alexey 2026-06-13). Архитектурный шов оставлен: единственная точка спауна в реестре —
место, куда потом встанет «лимит + политика вытеснения». Сейчас **не реализуется**.

### 3.4 Логирование (stdout/journald, как rj-kz4)

Логгер `renju.engine`. Префикс `game=<game_id> pid=<pid>` на каждой строке (через
`LoggerAdapter`/contextvar). До спауна — `pid=-`.

| Событие | Уровень | Поля |
|---|---|---|
| spawn | INFO | `game`, `pid`, `level_tag` |
| kill | INFO | `game`, `pid`, `reason` (leave/idle/shutdown/hang) |
| compute_move | INFO | `game`, `pid`, `moves`, `→ (x,y)`, `ms` |
| engine_invalid_move | WARNING | `game`, `pid`, `move`, `reason` (occupied/no-move) |
| forbidden_points | DEBUG | `game`, `pid`, `moves`, `n_forbidden` |
| hang/respawn/retry | WARNING | `game`, `pid`, `reason` |

**CWE-117:** `game_id`/`pid`/числа — без `safe()`; координаты ответа движка — int-кортежи
(инъекция структурно невозможна); `safe()` — только для сырой строки движка/ввода.

### 3.5 Протягивание game_id

`GameService` → `make_player(ctl, adapter, levels, game_id)` → `EnginePlayer` →
`game_service.engine_move(adapter, moves, params, game_id, level_tag)` →
`adapter.compute_move(game_id, moves, params, allowed_zone=zone, level_tag=...)`.
`forbidden_points` из `GameService.fouls` тоже принимает `game_id` и идёт в **тот же**
слот `game_id`, что и `compute_move` (один процесс на партию, §3.1 — не плодит второй).
HTTP/SSE-контракт ходов сигнатуры не трогают. `level_tag` извлекается в `make_player` из
`ctl.level_id` engine-оппонента (только для логов; на фолы не влияет — `_FORBID_PARAMS`
фиксированы). Дефолт `"-"` — для прямых обращений к реестру без engine-контекста (напр.
слот, поднятый фоновым `compute_move`/`forbidden_points`).

### 3.6 Эндпоинты присутствия (router → реестр)

- **`enter`** — при входе в партию, **одна точка на устройство-сессию**: на открытии
  игрового экрана `/game/:id`. Тонкий вызов в роутере → `registry.mark_present(game_id,
  level_tag)` (presence++, spawn если 0→1). `create_game` отдельный `enter` НЕ шлёт
  (иначе двойной presence у одного устройства → дрейф): создатель навигирует на экран и
  зашлёт enter оттуда. Если движку надо сходить раньше открытия экрана (человек-чёрный,
  фоновый `advance` сразу после create) — процесс поднимет сам `compute_move`
  (get-or-create при `presence==0`), enter с экрана потом переиспользует его.
- **`leave`** — **`POST /api/games/{id}/leave`** → `registry.mark_absent(game_id)`
  (presence--, release если 1→0 и нет idущего расчёта). Фронт зовёт на выходе на главный
  (rj-h0y).
- **SSE `/games/{id}/events`** — без изменений: **показывает ходы**, процессами не
  управляет.
- `release(game_id)` реестра — также API под реальное **удаление** партии (rj-as6).

> Дрейф `presence` (refresh без пары enter/leave, краш) допустим — процесс лишь дольше
> живёт, idle-sweep его добирает. Точный механизм enter (отдельный эндпоинт vs встроить
> в открытие) — деталь плана.

## 4. Что НЕ в этой задаче (scope-забор)

- **SSE-изоляция между играми** — уже есть (`event_hub` по `game_id`); SSE процессами не
  управляет. Менять/спекать нечего.
- **Персист/process-aware SSE/расширение модели данных** — не нужны: партия оживает из
  персистируемого состояния (`moves/status`), редкие корнеркейсы — `reset`+перезагрузка.
- **Лимит/LRU/насыщение** — нет (см. 3.3); шов оставлен, реализация — потом.
- **Движковый фикс бага аккумулятора при откате** — **rj-t95** (within-game undo), форк.
  Изоляция per-game его НЕ лечит. Полный фикс зевка = rj-899 + rj-t95.
- **Инкрементальное ведение партии, пул/предспаун, admin-UI процессов** — отдельно.

## 5. Зависимости

- **rj-t95** — нужен **в дополнение** к rj-899 (within-game undo).
- **rj-as6** — удаление партии использует `release()`.
- **rj-h0y / фронт** — вызвать `enter` на входе и `leave` на выходе (мелкая правка).

## 6. Открытые вопросы (только калибровка)

- **idle_timeout_s** (предложение 120–300 с), **sweep_interval_s** (≈30 с) — числа,
  калибруются на живой игре (как уровни), в настройки `RENJU_*`.
- **Лог User-Agent** на входе в партию (`renju.games`, `safe()`, PII) — для матчинга
  «устройство → game_id»? Опционально.

## 7. Критерий приёмки (черновой)

- Партия на двух устройствах → **один процесс, один pid** (2-е устройство переиспользует);
  respawn после hang меняет pid — это норма, логируется `reason=hang`.
- В логах по `game=<id>` видна жизнь процесса (spawn→ходы→kill) с pid.
- Параллельные ходы разных партий не сериализуются (общий `Lock` снят).
- **enter** (вход/создание) поднимает процесс; **leave** последнего устройства гасит;
  один `leave` при нескольких устройствах — не гасит.
- `leave` **не прерывает** идущий расчёт (compute-safety); release не делается сразу —
  слот реапит idle-sweep по простою.
- idle-таймаут реапит брошенный процесс (дрейф presence) по неактивности.
- Undo сохраняет процесс; после рестарта партия продолжается (реплей `moves`), модель
  данных не меняется.
- Регрессия rj-t95-репро (`moves[0:10]`, ждём `(12,7)`) — не ухудшилось (полный фикс
  зевка — после rj-t95).
