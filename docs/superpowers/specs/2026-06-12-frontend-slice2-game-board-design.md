# Фронт-срез 2: игровая доска (rj-p82) — дизайн

Дата: 2026-06-12. Тикет: `rj-p82` (эпик `rj-8wf`). Статус: зелёная после двух
независимых ревью; на утверждении Alexey.

Экран игры против движка: доска 15×15 по `prototype/index.html`, оптимистичный ход,
SSE-живость, undo, дебютные зоны, фолы, подсветка выигрышной пятёрки. Партия
создаётся временной кнопкой с захардкоженным уровнем — экран выбора уровня придёт
в срезе 3 (`rj-as6`).

Контекст: бэкенд-контур готов (этап 3 в `main`): `/api/games*`, SSE-хаб,
фоновый advance движка. Фронт-обёртка готова (срез 1): auth, Shell, роутинг,
конвенции стилей. Мастер-дизайн — `2026-06-07-renju-design.md` (§7, §8, §10).

## Решения (зафиксированы Alexey 2026-06-12)

- **Доска — DOM, не Canvas.** 225 статичных пересечений с редкими точечными
  изменениями — профиль DOM: клики, retina-чёткость, CSS-анимации и hover даром,
  доска тестируется RTL как обычный компонент (ориентир — chessground/lichess).
  Canvas оправдан тяжёлой графикой, которой нет и не планируется. Мастер-спека §7
  («Canvas/SVG») правится этим срезом; описание эпика rj-8wf — тоже.
- **Выигрышная линия подсвечивается, координаты считает бэк** (вариант B).
  Фронт правил не знает (тонкий клиент): домен уже детектирует пятёрку — он же
  вернёт её точки. Без миграции БД: линия вычислима из `moves` на лету.
- **Хардкод-уровень — `novice`**: движок отвечает мгновенно, ручное тестирование
  быстрое. Сила уровней калибруется позже (§4.5), для среза не важна.
- **Отказ сервера на ход = рассинхрон, не «причина».** Вся легальность отсечена
  на вводе; отказ достижим только при устаревшем состоянии (второе устройство,
  тихо умерший SSE) или сетевой ошибке. Обработка одна: откат камня → ресинхрон
  GET → нейтральное «Доска обновлена — ход не прошёл». Серверные проверки —
  подстраховка (и опора будущего PvP), фронт им сообщений не сочиняет.

  отказ сервера на undo обрабатывается как рассинхрон (та же доктрина);
- лог ходов (`.movelog`): «N. ⚫/⚪ H8», новые сверху.

## Контракт бэка глазами фронта

Эндпоинты (всё под cookie-auth из среза 1):

- `POST /api/games` body `{"opponent": {"kind": "engine", "levelId": "novice"}}` → `200` state.
- `GET /api/games/{id}` → `200` state; `404` — нет/чужая.
- `POST /api/games/{id}/move` body `{"x": 0..14, "y": 0..14}` → `202 {"accepted": true}`;
  `409` (`opponent_thinking`) / `422` (прочие причины) — для фронта равнозначны: рассинхрон.
- `POST /api/games/{id}/undo` → `200` state; `409`/`422` — рассинхрон.
- `GET /api/games/{id}/events?since={cursor}` → SSE-стрим.

State (ответ create/get/undo):

```json
{
  "id": "uuid", "owner_id": 1,
  "controllers": {"black": {"kind": "user"}, "white": {"kind": "engine", "levelId": "novice"}},
  "your_color": "black",
  "status": "awaiting_move | opponent_thinking | finished_black | finished_white | finished_draw",
  "moves": [[7,7], [8,8]],
  "undo_count": 0,
  "cursor": 5,
  "forbidden": [[5,8]],
  "winning_line": null
}
```

`winning_line` — **новое поле** (см. «Доработка бэка»): `[[x,y],…]` при
`finished_black/white`, иначе `null` (в т.ч. при ничьей).

SSE: события `event: {type}\ndata: {"seq": N, "type": "...", "payload": {...}}`;
heartbeat — комментарий `: ping` (EventSource его не видит). Типы:

| type | payload | Реакция фронта |
|---|---|---|
| `move` | `{"by": "black", "point": [x,y], "move_index": N}` | дорисовать камень / подтвердить pending (`move_index` абсолютный, центр = 0); обнулить `forbidden` — позиция сменилась, актуальный непустой набор придёт следом своим `forbidden`-событием (бэк публикует его до `status`) |
| `status` | `{"status": "...", "winning_line": [[x,y],…]?}` | обновить статус, блокировки, индикатор; при finished — подсветить линию. `forbidden`-набора это событие НЕ несёт |
| `forbidden` | `{"points": [[x,y],…]}` | заменить набор ✕ |
| `undo` | `{"move_count": K}` | усечь `moves` до K; статус → `awaiting_move`; снять pending, `winning_line`, `forbidden` |
| `error` | `{"message": "..."}` | ненавязчивое сообщение «Движок споткнулся — партия продолжится автоматически»; состояние не трогать. Доигрывание (§4.8) планируют GET партии и (пере)открытие `events` — штатный reconnect его и дёрнет |
| `reset` | `{}` | курсор недостижим → полный ресинхрон GET |

Семантика курсора: `cursor` монотонный per-game `seq`. Правило применения:
`seq ≤ cursor` — игнор (дубль реплея); `seq == cursor+1` — применить,
`cursor = seq`; `seq > cursor+1` — пропуск событий → полный ресинхрон GET
(после которого `cursor` берётся из state). После undo бэк события `forbidden`
с пустым набором не шлёт — поэтому сброс `forbidden` зашит в реакции на
`move`/`undo`, актуальный набор приходит отдельным `forbidden`-событием, когда
он непуст.

## Доработка бэка (внутри среза)

1. **Домен** (`app/domain/rules.py`): чистая функция
   `winning_line(moves: Sequence[Point]) -> list[Point] | None` — тем же
   сканом по линии последнего хода, что `outcome_after`, возвращает точки
   выигрышной серии в порядке вдоль направления (для белых при оверлайне — всю
   серию, может быть >5). Партия не завершена пятёркой (идёт/ничья) → `None`.
   Если последний ход замкнул две линии сразу — первая по порядку направлений
   `_DIRECTIONS` (детерминированно; для подсветки достаточно одной). Юнит-тесты:
   чёрная пятёрка (горизонталь/вертикаль/диагонали), белый оверлайн целиком —
   отдельно случай «последний ход в середине серии» (лучи в обе стороны),
   ничья → None, незавершённая → None, двойное замыкание → одна линия.
2. **Состояние** (`routers/games.py::_state`): поле
   `"winning_line": winning_line(moves) если status finished_black/white, иначе None` —
   вычисляется на лету, без миграции.
3. **События** (`game/service.py`): в обоих местах, где публикуется финальный
   `status` (ветка outcome в `advance`, финиш хода человека в `submit_move`),
   payload расширяется `"winning_line"` тем же вычислением. Нефинальные `status`
   поле не несут.
4. Тест контракта: партия доигрывается до пятёрки → state и `status`-событие
   несут одинаковую линию.

## Архитектура фронта

Новые файлы (конвенции среза 1: CSS-модули + `@value`-токены, инлайн только для
значений из данных; типы среза живут в `src/game/types.ts` — существующий
глобальный `src/types.ts` (тип `User`) остаётся как есть, не трогаем):

| Файл | Ответственность |
|---|---|
| `src/game/types.ts` | `GameState`, `GameEvent`, статусы — зеркало контракта |
| `src/game/api.ts` | `createGame(levelId)`, `getGame(id)`, `postMove(id,x,y)`, `postUndo(id)`, `getLevels()` поверх `apiRequest` |
| `src/game/reducer.ts` | **чистый** редьюсер `(state, event) → state \| "resync"`: применение SSE-событий, правило курсора, pending-подтверждение. Без I/O — ядро юнит-тестов |
| `src/game/legality.ts` | **чистая** легальность: клик — зеркало `opening_zone` (центр (7,7), квадраты 3×3/5×5 для ходов №1/№2), занятость, фолы, очерёдность (`color_to_move = moves.length % 2`), статус; undo-доступность (зеркало `undo_truncate`, см. UX). Отдаёт и зону для рамки |
| `src/game/useGame.ts` | хук: начальный GET, EventSource + reconnect, оптимистичный ход, undo, ресинхрон; отдаёт state + действия |
| `src/components/board/Board.tsx` + `.module.css` | гобан: сетка, хоси, камни, ghost, ✕, рамка зоны, маркер последнего хода, подсветка линии. Презентационный: props `{moves, forbidden, zone, winningLine, ghostColor, canPlay(x,y), onPlay(x,y)}` |
| `src/pages/GamePage.tsx` + `.module.css` | layout по прототипу: гобан + панель (карточки, undo, лог) |
| `src/test/eventsource.ts` | фейк EventSource для jsdom (вкл. имитацию разрыва) |

`HomePage` — временная кнопка. `App.tsx` — маршрут `/game/:id` (lazy, под
ProtectedRoute, в Shell).

**Оптимистичный ход**: клик легален → камень в `moves` сразу с пометкой pending
(внешне неотличим — сервер подтверждает за доли секунды), ввод заблокирован до
развязки → `POST move`. Подтверждение — своё `move`-событие: совпадение точки и
**абсолютного** `move_index` (центр = индекс 0, pending-камень уже лежит в
`moves`, его индекс тоже абсолютный). Гонка событие/ответ штатна: бэк публикует
`move` синхронно **до** возврата `202`, поэтому SSE-событие обычно приходит
раньше ответа POST — pending снимает **первый из двух исходов**, идемпотентно;
если pending уже подтверждён событием, результат POST (включая сетевую ошибку
ответа) игнорируется. Отказ POST (4xx) или сеть **до** подтверждения — снять
pending, `GET`-ресинхрон, сообщение «Доска обновлена — ход не прошёл». Разрыв
SSE при живом `202` pending переживает: после reconnect реплей с буфера донесёт
своё `move`-событие и снимет его. Pending одновременно максимум один.

**SSE-клиент** (`useGame`): первый `EventSource` открывается со
`since = state.cursor` **из GET-ответа** (не «после применения чего-либо»):
события, опубликованные в окне между GET и подпиской, бэк дореплеит с буфера —
окна потерь нет. Подписка на именованные типы. По `onerror`: закрыть, пауза 3 с, проверка сессии
`apiRequest GET /api/auth/me` (намеренно **без** `skipAuthRedirect` — отозванная
сессия уведёт на логин глобальным 401-обработчиком вместо вечного ретрая,
мастер-спека §10), затем новый EventSource с текущим курсором. Редьюсер вернул
"resync" → `GET /api/games/{id}` с заменой состояния и курсора. Размонтирование —
закрыть стрим.

## Геометрия и рендер (DOM)

- `.goban` — квадрат, фон-текстура `board_pine.jpg` (копируется в
  `src/assets/`), сетка и виньетка — слои из прототипа. Размер fluid:
  `width: min(100%, calc(100dvh - {шапка+отступы}))`, `aspect-ratio: 1` —
  квадрат к меньшей стороне вьюпорта, без JS.
- Все позиции — **в процентах от гобана** (инлайн из данных, конвенция среза 1):
  `pos(i) = PAD + i·STEP`, где `PAD = 30/592 ≈ 5.068%`, `STEP = 38/592 ≈ 6.419%`
  (пропорции прототипа). Константы геометрии — в одном модуле (`Board.tsx`).
  Размеры элементов (камень 32/592, ghost, хоси, маркеры) — в CSS-классах,
  тоже процентами от гобана.
- Слои z-index как в прототипе: сетка → хоси(2) → узлы-клики(3) → камни(4) →
  маркеры/✕(5). Анимация постановки камня — `@keyframes place` прототипа.
  Ghost виден на hover узла только если `canPlay(x,y)`; цвет — цвет ходящего.
- Сетка — `background-image: linear-gradient` с шагом в процентах
  (`calc(100%/14)`), позиционирована инсетом `PAD` — масштабируется вместе с
  гобаном; толщина линии — 1px-стоп градиента, как в прототипе (`.grid-lines`
  там добавляет `+1px` к габариту, чтобы замкнуть последнюю линию — учесть).

## Легальность ввода (зеркало домена)

Ввод разрешён ⇔ `status == awaiting_move` ∧ `color_to_move == your_color` ∧
нет pending. Точка кликабельна ⇔ не занята ∧ не в `forbidden` ∧ (зона активна →
точка в зоне). Зона: ход №1 (`moves.length == 1`) → 3×3 вокруг центра,
№2 → 5×5, дальше — без рамки и без ограничений. Это **зеркало** серверной
истины для UX; сервер остаётся последней инстанцией.

## Адаптив

По сквозным требованиям эпика: один desktop-лэйаут, fluid, CSS-only,
desktop-first. `≤900px` (`bpCompact`) — панель под доску
(`grid-template-columns: 1fr`, из прототипа). Гобан держит квадрат к
min-стороне вьюпорта (см. «Геометрия»). Тап-таргет узла = полный шаг сетки
(~38px при полном размере) — touch-friendly, hover-ghost — украшение, ввод от
него не зависит.

## Тестирование

- **Юниты (Vitest)**: `reducer.ts` — все типы событий, правило курсора (дубль /
  следующий / пропуск→resync), подтверждение pending, undo-усечение со сбросом;
  `legality.ts` — зоны по номеру хода, занятость, фолы, очерёдность, статусы,
  undo-доступность для обоих цветов (чёрные ≥ 3 ходов, белые ≥ 2).
- **RTL (jsdom + msw + фейк EventSource)**: клик по легальной точке → камень и
  POST; клик по нелегальной (занято/фол/вне зоны) → ничего; `move`-событие
  соперника → камень дорисован; отказ POST → камень снят и GET-ресинхрон;
  индикатор «думает» при `opponent_thinking` и блокировка ввода; финиш →
  надпись + подсветка линии; undo из завершённой партии → подсветка линии
  гаснет, статус «твой ход», ввод разблокирован; разрыв SSE → переоткрытие
  с курсором (вкл. кейс «pending подтверждается реплеем после reconnect»).
- **Бэк (pytest)**: юниты `winning_line`, контракт state+событие.
- **Ручная приёмка (Alexey)**: живая партия против novice через собранный
  `dist/` (включая Tailscale/https), скриншоты доски/зоны/финиша; направление
  нумерации лога и вид рамки зоны утверждаются здесь.
- Вне среза (по эпику): Playwright e2e-смоук и visual-regression отложены.
