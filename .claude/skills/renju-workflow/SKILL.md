# Renju Development Workflow

Actionable чек-лист процесса разработки Renju: pre-flight checks, развилка
«фича vs баг», recipe'ы для ревьюера, интеграция с beads как трекером.

**Why:** Alexey не разработчик, проверяет код плохо — но строго следит за
процессом. Пропущенная спека = недопонимание задачи = переделка → потеря
времени и доверия. Скрипт — не бюрократия, а страховка.

**Флоу — наш, не bd-шный.** bd = чистый трекер задач. Его предписания «как
работать» (session-close протокол, запреты на TaskCreate/MEMORY.md) НЕ
применяем — процесс задаёт этот скилл.

## Announce при активации

Сначала ОДНОЙ строкой: «Использую renju-workflow». Дальше — покажи Alexey
текущее состояние (ветка, `bd ready`, что есть/чего нет) и спроси, с какого
шага начинаем.

**Модели для subagent'ов:**
- **Разработка** (реализация, рефакторинг, тесты, type/lint cleanup) — `model: "sonnet"`.
- **Ревью** (спека, план, код) — `model: "opus"`. Opus лучше ловит слабые
  места, противоречия и edge cases — не разменивать эту силу на скорость.

## Beads (трекер задач)

- **Старт:** `bd ready` (доступное сейчас) · `bd show <id>` (детали) · `bd list --status=open`.
- **Взять в работу:** `bd update <id> --claim` ДО первой правки кода.
- **Нет задачи?** Создать ДО старта: `bd create --title="…" --description="…" --type=task|bug|feature --priority=2` (P0=critical … P4=backlog, числами).
- **Эпик:** `bd create` под-задачи + `bd dep add <issue> <depends-on>` для порядка (`bd ready` тогда показывает только разблокированное).
- **Discovered work** (нашли по ходу) → сразу `bd create`, не держать в голове.
- **После мержа:** `bd close <id> --reason="…"` — иначе доска копит in-progress.
- НЕ `bd edit` (открывает $EDITOR, блокирует агента).

## Развилка: фича или баг?

- **Фича / большой рефакторинг** → полный путь: `superpowers:brainstorming` →
  спека → ревью спеки → `superpowers:writing-plans` → ревью плана → реализация
  через `superpowers:subagent-driven-development` (или `superpowers:executing-plans`).
- **Баг / небольшой фикс с понятным поведением** → можно пропустить
  brainstorming и упростить спеку до 1-2 абзацев («что сломано, как должно быть,
  ограничения»), но **ревью спеки + план + ревью плана — всё равно**. При
  сомнении — спросить Alexey.
- **Эпик из нескольких задач** → одна общая спека + несколько под-планов /
  bd-под-задач с зависимостями.

## Pre-flight checks (перед стартом)

До любой правки кода:

1. **bd-задача.** `bd ready` / `bd show <id>`. Нет — создать. `bd update <id> --claim`.
2. **Ветка.** `git branch --show-current`. Если `main` — **СТОП**, создать feature/chore-ветку.
3. **Чистый main.** `git status --short`. Хвосты — закоммитить или спросить Alexey
   перед веткой. Особо: `.beads/issues.jsonl` (bd его трогает — легко забыть).
4. **Спека есть?** `docs/superpowers/specs/`. Нет — `superpowers:brainstorming`
   (фича) или draft (баг). Источник истины дизайна — `2026-06-07-renju-design.md`.
5. **План есть?** `docs/superpowers/plans/`. Нет — `superpowers:writing-plans` после фикса спеки.

## Recipe: запуск ревьюера

Ревьюер — subagent, строго **READ-ONLY** (читает, может гонять тесты для
проверки, но код НЕ правит — чинит только implementer). После каждого
артефакта (спека, план, код):

```
Agent({
  subagent_type: "general-purpose",
  model: "opus",
  description: "Ревью спеки/плана/кода",
  prompt: "Прочитай <путь>. Найди ВСЕ проблемы: unclear scope, missing edge
  cases, нарушения правил проекта/CLAUDE.md, breaking changes, missed tests.
  Каждая находка: серьёзность (blocker/major/minor), файл:строка, что не так,
  что предложить. READ-ONLY: НЕ исправляй сам, НЕ запускай тесты как задачу —
  только findings."
})
```

**ОБЯЗАТЕЛЬНО включай буквальную цитату секции «Что НЕ в этом этапе» / «Вне
области задачи» из спеки/плана.** Без неё ревьюер выдаёт по делу-корректные
предложения из scope соседнего тикета. Формулировка: «Считай следующее **за
scope этой задачи** (не предлагай как findings): <буквальный текст секции>».

## Цикл ревью: фикс → ре-ревью → до зелёного

Ревью — **НЕ один проход** и **НЕ тот же ревьюер.** После того как implementer внёс
правки по findings, запусти **СВЕЖЕГО, независимого ревьюера** (новый subagent, **НЕ
`resume` прежнего**): дай ему артефакт + список прошлых findings (сверить, что закрыты)
и попроси **независимо пересмотреть целиком** — поймать и баги, внесённые самими
правками, и то, что прошлый проход пропустил. Возобновлённый тот же агент лишь
проверяет «закрыл ли ты МОИ findings»: он вложен в свой вердикт и слеп к своим
промахам — это смазывает независимость, в которой весь смысл ревью (поймали 2026-06-12).
Крути **фикс ↔ свежее ре-ревью, пока не GREEN.** Автономный цикл — **Alexey в каждый
круг НЕ дёргается.**

**Обработать надо ВСЕ findings** (FIX или явный KEEP с обоснованием) — ничего не
пропускать. Но **предъявлять Alexey — только то, что меняет архитектуру /
поведение / UI**, и обязательно **с контекстом** (что в коде сейчас, что изменится
в поведении/риске — Alexey не разработчик, без контекста решить не может).
Остальное (формулировки, фактические правки описания, добавление/перенос тестов,
переименования без смены поведения) — **чинить молча**, в тот же цикл до зелёного,
не вынося на стол и не подсовывая как гейт.

Alexey подключается, когда ревьюер **зелёный**: финальное «добро» по
арх/поведенческим решениям и утверждение артефакта. Только после его зелёного —
следующий шаг (план / реализация / мерж). Не норови перепрыгнуть на след. фазу,
не доведя текущую до зелёного.

## Чек-лист (16 шагов)

Строго по порядку. После каждого артефакта — ревью субагентом, затем цикл
**фикс ↔ ре-ревью до зелёного** (см. «Цикл ревью»). ВСЕ findings обрабатываются
(FIX/KEEP); Alexey предъявляются только арх/поведение/UI (с контекстом).

| # | Шаг | Кто / чем |
|---|---|---|
| 1 | Спека (design-doc) — `superpowers:brainstorming` для фичи | Claude |
| 2 | Ревью спеки → ВСЕ findings | Subagent (Opus) + Alexey |
| 3 | Фикс спеки + утверждение | Claude + Alexey |
| 4 | План — `superpowers:writing-plans` (TDD-детализация) | Claude |
| 5 | Ревью плана → ВСЕ findings | Subagent (Opus) + Alexey |
| 6 | Фикс плана + утверждение | Claude + Alexey |
| 7 | Feature branch от main + `bd update <id> --claim` | Claude |
| 8 | **Тесты ПЕРЕД реализацией** (TDD: red) — `superpowers:test-driven-development` | Subagent (Sonnet) |
| 9 | Реализация → тесты green — `superpowers:subagent-driven-development` | Subagent (Sonnet) |
| 10 | Ручное тестирование (`play_cli` против живого движка) | **Alexey** (ждать!) |
| 11 | Коммит **после** одобрения Alexey | Claude |
| 12 | Спек-ревью кода (соответствие спеке) | Subagent (Opus) + Alexey |
| 13 | Код-ревью (качество) — `superpowers:requesting-code-review` | Subagent (Opus) + Alexey |
| 14 | Фикс ВСЕХ findings + повторный прогон тестов | Claude |
| 15 | Мерж — **только** по явной команде | Alexey говорит «мержи» |
| 16 | Пуш — **только** по явной команде (если есть remote) | Alexey говорит «пуш» |

**Шаги 12–13 — ОТДЕЛЬНЫЙ полный холистик-ревью по ВСЕЙ ветке** (интегрированный
результат целиком), **СВЕЖИМ независимым** ревьюером, **перед итоговой приёмкой/мержем.
НИКОГДА не сворачивать в по-задачные** — те видят дифф каждой задачи в отдельности и
пропускают межзадачные / интеграционные / архитектурные стыки (поймали 2026-06-12:
контракт async-`202` был реализован inline — по-задачные ревью не заметили, свежий
холистик заметил).

**После мержа (всегда):** `bd close <id> --reason="…"`. Cleanup, но обязательный.

## Жёсткие запреты

- **Не работать в main.** Сначала feature-ветка.
- **Не прыгать на код без спеки + плана.** Даже «простая» задача: спека/план — 10 минут, спасают часы.
- **TDD: тесты ПЕРЕД кодом.** Самое частое нарушение. «Допишу тесты после» — стоп, переписать.
- **Findings — ВСЕ обрабатываются** (FIX или явный KEEP с обоснованием), ничего не
  пропускать. Но **предъявлять Alexey — только меняющее арх/поведение/UI** (с
  контекстом); остальное чинить молча. После любых правок — **ВСЕГДА ре-ревью до
  зелёного** (см. «Цикл ревью»), не подсовывать сырые после-правочные изменения как гейт.
- **НИКОГДА не пропускать находку отмазкой** «это вне скоупа / это отложим / это не я
  писал» — **и «это безвредно / не-баг / подтверждено как ок» — ТОЖЕ отмазка**, если за
  ней нет действия. Главная идея ревью — **не пропустить НИЧЕГО.** КАЖДУЮ находку либо
  **ЧИНИШЬ**, либо, если она и правда не баг, — **конкретно проверяешь** (трассируешь код,
  а не «на глаз») **И вписываешь обоснование В АРТЕФАКТ** (заметка в плане/спеке/коде:
  почему безопасно — чтобы на это не наступили снова). «Безвредно» как устный вердикт без
  записи и без действия = ПРОПУСК. Скопом «находка X и ниты подтверждены безвредными» —
  запрещено: разбирать поштучно. (Поймали 2026-06-12.) Отсрочка легитимна ТОЛЬКО как
  осознанное, задокументированное решение, **предъявленное Alexey** (тикет + причина) —
  не молчаливый wave-off. Scope-забор исключает из findings лишь СОЗНАТЕЛЬНО отложенное
  в конкретный тикет, а не «область, которую можно игнорировать».
- **Полный холистик-ревью по ВСЕЙ ветке — ВСЕГДА перед итоговой приёмкой** (шаги
  12–13), отдельным проходом, **СВЕЖИМ независимым** ревьюером. Не сворачивать в
  по-задачные (они слепы к межзадачным/интеграционным/архитектурным стыкам).
- **Ре-ревью — СВЕЖИМ ревьюером, не `resume` прежнего.** Возобновлённый агент проверяет
  свою же домашку и слеп к собственным промахам — это не независимая проверка.
- **Ревьюеры строго read-only.** Код правит только implementer. Ревьювер, начавший
  писать/выполнять код — нарушение роли, прерывать.
- **Pre-existing findings не «откладывать».** Нашли старое — фиксим в том же тикете.
- **Мерж без явной команды Alexey запрещён.** «Окей», «ок», «продолжай», «ага» —
  **не команды**. Команда — «мержи», «мерж».
- **Пуш — отдельная команда от мержа.** Мерж ≠ пуш.
- **Pytest — последовательно, не параллельно.** Backend unit + integration с живым
  движком: shared state (процесс Rapfi, temp-файлы) даёт ложные failures.

## Когда отойти от скрипта

Тривиальная правка — **тайпо в комменте, переименование одной локальной
переменной, форматирование, правка config/hook/spec**. Не «маленький
рефакторинг», не «всего три строки», не «явно понятно что делать». Хочешь
пропустить шаги — **спроси Alexey прямо**: «через workflow или одноразовым
фиксом?». Не решай сам.

При сомнении — `superpowers:brainstorming` короткой формой (1-2 минуты) лучше,
чем «прыгнул в код, переделал три раза».
